# 第8章 质量属性

## 8.1 质量属性目标

本章定义产研军团OS在不同演进阶段必须满足的系统级质量要求。质量属性是架构设计的验收标准，也是技术选型的约束条件。

**核心原则：**
- MVP阶段满足"可用"底线，不追求极致
- 成长期逐步提升，支撑规模化运行
- 成熟期达到企业级标准

---

## 8.2 性能

### 8.2.1 响应时间

| 场景 | MVP目标 | 成熟期目标 | 说明 |
|------|---------|-----------|------|
| 页面首次加载 | <3s | <1.5s | 含BFF聚合查询 |
| 任务列表查询 | <500ms | <200ms | 含分页+筛选 |
| 工作流状态查询 | <500ms | <200ms | 含当前节点+历史 |
| 价值积分计算 | <5s（异步） | <2s（异步） | 批量计算，非实时 |
| 数字员工任务分配 | <2s | <500ms | 含规则匹配+实例创建 |
| 跨服务调用（P99） | <500ms | <200ms | OpenFeign同步调用 |

### 8.2.2 吞吐量

| 指标 | MVP目标 | 成熟期目标 |
|------|---------|-----------|
| 并发在线用户 | 50 | 500 |
| 日活任务操作 | 1,000 | 50,000 |
| 事件处理速率 | 100/s | 2,000/s |
| 数字员工并发执行 | 5 | 50 |

### 8.2.3 资源消耗

| 指标 | MVP目标 | 成熟期目标 |
|------|---------|-----------|
| 单服务内存 | <1GB | <2GB |
| 单服务CPU | <1核 | <2核 |
| 数据库连接池 | <20/服务 | <50/服务 |
| 单次LLM调用token | 按实际 | 按实际 |

---

## 8.3 可用性

### 8.3.1 系统可用性

| 阶段 | 目标 | 说明 |
|------|------|------|
| MVP | 99.5% | 允许计划内停机部署，无多副本 |
| 成长期 | 99.9% | 核心服务多副本，滚动更新 |
| 成熟期 | 99.95% | 全链路高可用，自动故障转移 |

### 8.3.2 故障恢复

| 场景 | MVP策略 | 成熟期策略 |
|------|---------|-----------|
| 单服务宕机 | 手动重启 | 自动重启+流量切换 |
| 数据库宕机 | 人工介入恢复 | 主从切换（<30s） |
| 消息队列堆积 | 告警+人工清理 | 自动扩消费端 |
| 数字员工实例故障 | 标记失败，人工重试 | 自动重试3次，失败后转人工 |

### 8.3.3 数据持久化

- 核心业务数据（项目、任务、价值积分）实时持久化
- 事件日志至少保留30天
- 数据库每日全量备份

---

## 8.4 安全性

### 8.4.1 认证与授权

| 要求 | MVP | 成熟期 |
|------|-----|--------|
| 用户认证 | JWT + 用户名密码 | JWT + SSO集成 |
| 数字员工认证 | API Key | mTLS证书 |
| 权限粒度 | 角色级 | 数据范围级 |
| 会话超时 | 8小时 | 可配置 |

### 8.4.2 数据安全

| 要求 | 实现 |
|------|------|
| 传输加密 | HTTPS全链路 |
| 敏感字段加密 | 数据库层AES加密（API Key、Token） |
| 多组织隔离 | 数据库行级隔离（org_id字段） |
| 审计日志 | 关键操作全记录（谁、何时、做了什么） |

### 8.4.3 数字员工安全

- 数字员工API Key定期轮换（默认90天）
- 数字员工操作范围受限于任务授权
- 数字员工访问外部系统需审批

---

## 8.5 可扩展性

### 8.5.1 服务扩展

| 维度 | MVP | 成熟期 |
|------|-----|--------|
| 服务数量 | 4+1聚合 | 12独立服务 |
| 数据库 | 1核心库+1用户库 | 9独立库 |
| 水平扩展 | 不支持 | 核心服务支持 |
| 数字员工运行时 | 远程API | 容器化+k8s调度 |

### 8.5.2 功能扩展

- 新工作流节点类型：通过配置注册，无需改代码
- 新价值积分规则：通过规则引擎配置
- 新外部系统集成：通过基础设施域接口配置

---

## 8.6 可维护性

### 8.6.1 监控与可观测性

| 维度 | MVP | 成熟期 |
|------|-----|--------|
| 日志 | 文件日志 | ELK聚合 |
| 指标 | 基础JVM指标 | Prometheus+Grafana |
| 链路追踪 | 无 | SkyWalking |
| 告警 | 企业微信通知 | 分级告警+自动处理 |

### 8.6.2 部署与升级

| 要求 | MVP | 成熟期 |
|------|-----|--------|
| 部署方式 | Docker Compose | k8s Helm |
| 升级方式 | 停机部署 | 滚动更新 |
| 配置管理 | 配置文件 | Nacos配置中心 |
| 版本回滚 | 手动 | 一键回滚 |

---

## 8.7 兼容性

### 8.7.1 API兼容性

- REST API：URL版本控制（/v1/、/v2/）
- 事件格式：新增字段兼容，废弃字段保留至少2个版本
- 数据库Schema：支持增量迁移，不可逆变更需审批

### 8.7.2 外部系统集成

- GitLab/Jira/企业微信：支持Webhook回调
- CI/CD系统：支持标准API调用（Jenkins/GitLab CI）
- LLM服务：支持多提供商切换（OpenAI/Claude/本地模型）

---

## 8.8 质量属性验证方法

| 质量属性 | 验证方法 | 验收标准 |
|----------|---------|---------|
| 性能 | 压测（JMeter/k6） | P99响应时间达标 |
| 可用性 | 故障注入（Chaos Monkey） | 故障恢复时间达标 |
| 安全性 | 渗透测试 + 代码审计 | 无高危漏洞 |
| 可扩展性 | 扩容测试 | 吞吐量线性增长 |
| 可维护性 | 部署演练 | 升级时间<30分钟 |

---

## 8.9 与第6章的关联

本章质量属性是第6章架构设计的验收标准：
- 6.1 架构约束 → 本章8.2-8.7的具体量化目标
- 6.9 非功能性需求 → 本章8.2性能 + 8.3可用性的细化
- 6.15 监控与告警 → 本章8.6.1的可观测性要求

> **MVP阶段底线：** 系统可用、数据安全、性能可接受。不追求高可用、不追求极致性能、不追求全自动运维。
