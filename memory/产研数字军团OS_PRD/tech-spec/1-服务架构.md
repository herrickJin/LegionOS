## 2. 服务清单

| 服务名 | 服务ID | 端口 | 包含域 | 数据库 | 说明 |
|--------|--------|------|--------|--------|------|
| **pdm-svc** | core-service | 8080 | 经营决策+项目管理+项目实施+产研流程 | core_db(MySQL) | 高频联动，共享Schema隔离 |
| **asset-svc** | asset-service | 8081 | 技术资产+知识经验+自我进化+价值评估 | asset_db(MySQL)+MongoDB | 辅助域聚合 |
| **agent-svc** | agent-service | 8082 | 数字员工 | agent_db(MySQL)+Redis | 远程LLM API调用 |
| **portal-svc** | portal-service | 8083 | 个人域(BFF) | 无 | 只读聚合 |
| **iam-svc** | iam-service | 8084 | 用户权限 | iam_db(MySQL) | 独立，RBAC权威源 |

## 4. 数据架构

### 4.1 数据库实例

```
MySQL实例（单实例，多Schema/库）：
├─ core_db
│   ├─ opp_schema（经营决策）
│   ├─ prj_schema（项目管理）
│   ├─ tsk_schema（项目实施）
│   └─ wf_schema（产研流程）
├─ iam_db（用户权限）
├─ asset_db（技术资产+知识经验+自我进化+价值评估）
└─ agent_db（数字员工）

MongoDB实例：
├─ opp_db（MRD文档）
├─ knw_db（知识文档）
└─ evo_db（改进需求）

Redis：
├─ 用户缓存（iam数据本地缓存）
├─ 数字员工运行时状态
└─ 项目上下文
```

### 4.2 Schema隔离规则

- core_db内4个Schema，各服务独立数据库用户
- 禁止跨Schema直接访问，通过服务层API交互
- 所有业务表含 `org_id` 字段，应用层过滤

## 6. 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 服务聚合 | MVP 4+1 | 减少跨服务调用，降低Week 1搭建复杂度 |
| 数据库 | 单MySQL实例+Schema隔离 | 本地事务保证ACID，避免分布式事务 |
| 事件总线 | RabbitMQ（无死信队列） | MVP失败人工处理，降低运维复杂度 |
| 数字员工 | 远程LLM API调用 | 不自研运行时，Week 2可快速接入 |
| 网关权限 | Gateway做完整校验 | 服务层不复验，只过滤数据范围 |
| 缓存 | Redis本地缓存+TTL | 用户数据缓存1小时，事件触发刷新 |
