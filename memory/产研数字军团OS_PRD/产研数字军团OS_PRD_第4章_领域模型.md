# 产研数字军团OS PRD

## 4. 领域模型

---

### 4.1 领域划分

产研数字军团OS 划分为 **12 个业务域**：

| 域            | 类型  | 核心实体                           | 职责                                                                          |
| ------------ | --- | ------------------------------ | --------------------------------------------------------------------------- |
| **经营决策域**    | 核心  | 商机、可行性报告、MRD                   | 从洞察到立项决策，产出MRD；资源申请仅提供MRD文档，汇报和申请流程在OS外                                     |
| **项目管理域**    | 核心  | 项目、里程碑、迭代、资源、变更记录、风险项          | 接收MRD后立项，围绕项目立项、变更、监管、资源调配                                                  |
| **项目实施域**    | 核心  | 任务、执行计划、交付物、进度                 | 任务执行、协调、跟踪；任务分配由产研流程域驱动，本域负责任务实例生命周期                                        |
| **产研流程域**    | 核心  | 任务规范、任务实例、节点定义、判定规则、执行工作台、执行评价 | 定义任务由谁执行（人/数字员工），核心产出任务规范；对人和数字员工一视同仁；任务FAQ由知识经验域提供，本域负责关联；包含9个典型任务规范的工作台定义 |
| **数字员工域**    | 核心  | 数字员工原型、数字员工实例、灵魂、Body定义、部署规格   | 数字员工定义、登记、发布、监管；Body逻辑定义由本域管理，物理运行由基础设施域承载                                  |
| **用户权限域**    | 支撑  | 用户、角色、权限、组织                    | 用户体系、RBAC权限、组织层级；数字员工也是用户，功能权限由本域管理，业务能力由数字员工域灵魂配置决定                        |
| **技术资产域**    | 核心  | 技术资产、资产库、复用记录、版本               | 技术资产的提供、积累、管理                                                               |
| **知识经验域**    | 核心  | 知识条目、知识库、分类标签、版本历史             | 知识经验的提供、积累、管理；提供任务FAQ，供产研流程域关联                                              |
| **基础设施域**    | 支撑  | 部署节点、Body实例、项目上下文、接口服务         | 数字员工Body和项目上下文的运行时环境，提供对外接口和服务                                              |
| **自我进化域**    | 辅助  | 改进需求、跟踪记录                      | 记录系统级改进需求，工程师在OS外优化，结果回流记录；解决后可触发价值积分计算                                     |
| **价值评估与兑现域** | 辅助  | 价值规则、价值积分、评估报告、兑现记录            | 为经营决策者评估人、数字员工、技术资产、知识经验提供参考                                                |
| **个人域**      | 视图  | 我的项目、资产、待办、消息、通讯录              | 以用户个人视角聚合跨域数据，提供个性化工作台；**只读聚合层**，业务数据修改需回源到对应域                              |

### 4.3 核心域详解

#### 4.3.1 经营决策域

```
商机 ──评估──→ 可行性报告 ──决策──→ MRD
 │                                │
 ▼                                ▼
需求输入                     项目（进入项目管理域）
```

**核心实体：** 商机、可行性报告、MRD

**状态流转：**
- 商机：发现 → 评估中 → 可行/不可行 → 产出MRD/归档
- MRD：草稿 → 评审中 → 已通过/已拒绝 → 已归档

**关键设计：**
- 资源申请仅提供MRD文档，汇报和申请流程在OS外完成
- 经营决策域不创建项目，项目由项目管理域创建
- 商机评估本身不产生价值积分，价值积分从项目创建后的任务执行开始计算
- MRD"已通过"后移交项目管理域，经营决策域不再跟踪；已通过和已拒绝的MRD都进入"已归档"

---

#### 4.3.2 项目管理域

```
项目 ──分解──→ 里程碑 ──拆解──→ 任务（进入项目实施域）
 │
 ├── 创建项目（接收MRD后创建，或OS外直接创建）
 ├── 资源调配
 ├── 进度监管
 ├── 变更管理
 └── 风险预警
```

**核心实体：** 项目、里程碑、迭代、资源、变更记录、风险项

**状态流转：**
- 项目：已创建 → 启动 → 进行中 → 交付中 → 收口 → 归档
- 里程碑：规划中 → 进行中 → 已达成 → 已延期
- 迭代（可选，敏捷模式）：规划中 → 进行中 → 已完成
- 资源：已申请 → 已分配 → 已释放

**关键设计：**
- 项目可由MRD驱动创建，也可由OS外直接驱动创建
- 风险项状态：已识别 → 评估中 → 处理中 → 已缓解 → 已关闭（或已接受）
- 里程碑可关联任务规范的完成作为达成条件
- 里程碑与迭代互斥使用：里程碑适用于瀑布模式，迭代适用于敏捷模式
- 变更记录包含：变更类型、变更原因、影响评估、审批状态、执行结果

---

#### 4.3.3 项目实施域

```
任务 ──分配──→ 执行者（人/数字员工）──执行──→ 交付物
 │                                              │
 ├── 执行计划                                    ├── 代码
 ├── 进度跟踪                                    ├── 文档
 ├── 阻塞协调                                    ├── 测试结果
 └── 验收                                        └── 发布包
```

**核心实体：** 任务、执行计划、交付物、进度、阻塞项、验收记录

**状态流转：**
- 任务：待分配 → 待开始 → 进行中 → 待审查 → 已完成 → 已验收

**关键设计：**
- 阻塞项是任务执行层面的即时障碍，由本域处理；若无法及时解决且影响项目目标，可升级为项目管理域的风险项
- 任务分配由产研流程域的判定规则驱动，本域只负责任务实例的生命周期管理
- 验收结果：通过、不通过、有条件通过（需返工）
- 交付物状态：草稿→已提交→已验收→已归档

---

#### 4.3.4 产研流程域

```
任务规范类型 ──定义──→ 任务规范 ──编排──→ 任务实例
    │                       │                  │
    ├── 项目规范            ├── 元数据         ├── 节点执行序列
    ├── 普通任务规范        ├── 输入/输出定义  ├── 状态流转
    └── 子任务规范          ├── 验收标准       └── 交付物关联
                            ├── 节点编排
                            ├── 判定规则
                            └── 工作台定义
```

**核心实体：** 任务规范、任务实例、节点定义、判定规则、执行工作台、执行评价

**任务规范6要素：**

| 要素 | 说明 | 示例 |
|------|------|------|
| **1. 元数据** | 规范标识、名称、版本、类型、分类、标签、Owner、状态 | "Java代码开发任务_v1.2" |
| **2. 输入项与输入标准** | 执行前需要的数据、格式要求、完整性校验 | 需求文档(ID)、接口定义、设计稿链接 |
| **3. 产出项与验收标准** | 执行后产出的交付物、质量门槛、验收条件 | 代码PR、单元测试报告、覆盖率≥80% |
| **4. 子任务规范清单+任务流** | 嵌套的子任务规范（内联或引用）、执行顺序、依赖关系 | 接口定义→业务逻辑→单元测试 |
| **5. 执行岗责任要求** | 执行者类型（人/数字员工）、技能要求、判定规则、责任边界 | 技能：Java L3+SpringBoot；负载<3 |
| **6. 执行工作台** | 任务执行时的界面：定制UI或规范说明页 | 轻量IDE面板/Review面板/测试面板 |

**关键设计：**
- **任务规范是OS的核心抽象**，定义"什么人在什么条件下执行什么动作、产出什么、如何验收"
- **项目规范是任务规范的一种类型**，由MRD触发，包含多个普通任务规范
- 普通任务规范可独立执行，由项目规范或人工触发
- 子任务规范嵌套在普通任务规范内，不可独立触发
- 子任务规范支持**混合模式**：内联定义（仅本任务使用）或引用外部规范（公共规范复用）
- 对人和数字员工一视同仁，都需满足执行岗责任要求中的技能成熟度要求
- 判定规则明确"谁更适合执行"，而非硬性指定执行者
- 任务支持人节点、数字员工节点、审核节点、自动化节点的混合编排（MVP只支持线性）
- 任务必须关联任务规范，无规范不允许发布
- 产研流程域与项目实施域形成反馈闭环：规范定义 → 任务执行 → 效果评价 → 规范优化
- 任务FAQ由知识经验域提供，本域负责关联
- 技能成熟度是硬性能力评级（L1-L5），经验要求是软性要求，两者共同构成筛选条件

```
│   ├── 设计类型标签（需求/架构/概要/详细/专项/验证）
│   ├── 关联MRD/需求文档入口
│   └── 版本历史下拉框
├── 左侧：模板与组件库
│   ├── 模板区（按设计类型筛选）
│   │   ├── 业务流程图模板（T9.1）
│   │   ├── 需求矩阵模板（T9.2）
│   │   ├── **交互原型设计模板（T9.3）**
│   │   ├── 架构图模板（T10）
│   │   ├── 模块划分模板（T11）
│   │   ├── 类图模板（T12）
│   │   ├── 安全 checklist（T13）
│   │   └── 评审报告模板（T14）
│   └── 组件库（可拖拽）
│       ├── 流程图组件
│       ├── 架构图组件
│       ├── UML组件
│       ├── **原型组件（按钮/表单/卡片/导航）**
│       └── 自定义组件
├── 中部：画布编辑区
│   ├── 无限画布（支持缩放/平移）
│   ├── 拖拽放置组件
│   ├── 连线与关系标注
│   └── 实时协作光标（多人编辑）
├── 右侧：属性与AI辅助
│   ├── 属性面板（选中组件的属性编辑）
│   ├── AI辅助面板
│   │   ├── "帮我生成模块划分"
│   │   ├── "检查架构合理性"
│   │   └── "生成接口定义"
│   └── 关联文档（需求/MRD/技术规范）
└── 底部：操作栏
    ├── 保存草稿
    ├── 版本对比
    ├── 导出（PNG/PDF/Markdown）
    ├── 提交评审（T14）
    └── 标记完成
```

**工作台设计原则：**
- 高频协作任务（Review、文档、设计）→ 全流程OS内完成
- 编码类任务 → OS内浏览+上下文+提交，编码在本地IDE
- AI分析类任务 → 数字员工自动执行，OS内展示结果+人工确认
- 设计类任务 → OS内模板引导+画布编辑+AI辅助，设计文档OS内生成
- 所有任务 → 统一入口：任务列表点击 → 打开对应工作台
- **日志分析类任务 → 数字员工自动分析日志文件，OS内展示异常事件+根因+修复建议，人工确认后归档**

#### 4.3.5 数字员工域

```
数字员工原型 ──实例化──→ 数字员工实例 ──分配──→ 项目/任务
    │                        │
    ├── 灵魂                 ├── 灵魂（继承原型）
    │   ├── 思维模式          ├── Body（动态分配）
    │   ├── 知识库            ├── 项目Memory
    │   ├── 技能配置          └── 任务上下文
    │   └── 行为准则
    └── 部署规格说明书
```

**核心实体：** 数字员工原型、数字员工实例、灵魂、Body定义（含环境检测脚本、部署脚本）、部署规格

**状态流转：**
- 原型：设计中 → 测试中 → 已发布 → 已下线/已废弃
- 实例：创建中 → 待分配 → 执行中 → 闲置 → 已回收

**状态说明：**
- 已下线：暂时停用，可重新上线
- 已废弃：永久淘汰，不可恢复

**关键设计：**
- Body定义是抽象配置（环境检测脚本、部署脚本），基础设施域负责物理运行
- 灵魂的知识库是数字员工私有工作记忆，区别于知识经验域的组织级共享知识库
- 实例闲置超阈值后自动回收，回收前需确认无未完成任务
- 项目Memory是数字员工域维护的语义记忆，区别于基础设施域提供的项目上下文（环境配置数据）
- 思维模式和行为准则由AI Agent架构师在原型定义时配置
- 实例默认继承原型的灵魂配置，实例级可覆盖知识库、技能配置、行为准则；思维模式不允许实例级覆盖

#### 4.3.6 技术资产域

```
技术资产 ──沉淀──→ 资产库 ──检索──→ 复用
    │                  │
    ├── 公共组件       ├── 按类型
    ├── 系统框架       ├── 按技术栈
    ├── 工具脚本       ├── 按项目
    ├── 自动化流水线   └── 按关键词
    └── 工具软件
```

**核心实体：** 技术资产、资产库、复用记录、版本

**状态流转：** 待审核 → 已收录 → 已发布 → 已废弃

**关键设计：**
- 资产复用行为产生事件，推送到价值评估域计算积分
- 技术资产采用语义化版本（SemVer），版本升级触发审核流程
- 技术资产与资产库是多对多关系
- 资产库由AI Agent架构师或复合型研发经理创建，技术资产收录需审核

#### 4.3.7 知识经验域

```
知识经验 ──沉淀──→ 知识库 ──检索──→ 应用
    │                  │
    ├── 规范文档       ├── 按领域
    ├── 方法论         ├── 按场景
    ├── 踩坑记录       ├── 按角色
    ├── 决策记录       └── 按关键词
    └── 问答
```

**核心实体：** 知识条目、知识库、分类标签、版本历史

**状态流转：** 草稿 → 审核中 → 已发布 → 已更新 → 已归档

**关键设计：**
- 知识条目被引用/应用时产生事件，推送到价值评估域计算积分
- 分类标签属于当前版本的知识条目，不保留历史标签；版本历史只记录内容变更，标签随当前版本更新

#### 4.3.8 基础设施域

```
部署节点 ──承载──→ Body实例 ──运行──→ 数字员工灵魂
    │                  │
    ├── 计算资源       ├── 项目上下文加载
    ├── 存储资源       ├── 外部系统接口
    ├── 网络资源       └── 日志与监控
    └── 安全隔离
```

**核心实体：** 部署节点、Body实例、项目上下文、接口服务

**部署节点状态：** 在线 → 满载 → 维护中 → 离线

**关键设计：**
- Body实例由基础设施域承载运行，数字员工域只管理Body逻辑定义
- 为项目实施域提供项目上下文（git地址、访问数据、共享目录等）
- 项目上下文的定义由项目管理域/项目实施域负责，本域仅负责承载和隔离
- Body可按负载动态扩缩容
- 对外提供三类接口：Body管理接口、项目上下文接口、资源监控接口
- 外部系统接口（GitLab/Jira/企业微信等）由AI Agent架构师配置，供数字员工实例调用

#### 4.3.9 自我进化域

```
改进需求 ──收集──→ 分类整理 ──跟踪──→ 解决/关闭
    │                      │
    ├── 数字员工优化        ├── 待处理
    ├── 产研流程改进        ├── 处理中
    ├── 系统功能增强        └── 已验证
    └── 体验问题
```

**核心实体：** 改进需求、分类标签、处理状态、解决方案

**关键设计：**
- 只跟踪系统级改进需求，项目级问题留在项目管理域
- 工程师在OS外寻找解决方案，结果回流到本域记录
- 不直接驱动变更，只提供跟踪和参考
- 改进需求解决后可触发价值评估域积分计算（非强制）
- 体验问题可快速修复的由AI Agent架构师直接处理并记录；需深入分析的按标准改进需求流程处理
- 解决方案状态：草稿→已提交→已验证→已采纳/已拒绝

#### 4.3.10 价值评估与兑现域

```
价值规则 ──计算──→ 价值积分 ──汇总──→ 评估报告
    │                      │
    ├── 人                 ├── 个人价值报告
    ├── 数字员工           ├── 团队价值报告
    ├── 技术资产           └── 资产价值报告
    └── 知识经验
```

**核心实体：** 价值规则、价值积分、评估报告、兑现记录

**兑现记录关键字段：**
- 兑换类型：资源 / 激励
- 兑换状态：申请中 / 已批准 / 已拒绝 / 已完成
- 关联价值积分变动记录

**关键设计：**
- 经营管理者可消耗积分兑换资源，兑换行为在OS内记录，实际资源交付在OS外完成
- 积分来源：任务执行、资产复用、知识沉淀、系统改进贡献
- 价值规则由经营管理者或AI Agent架构师配置，支持按域、按角色、按时间维度自定义权重
- 兑换申请提交时冻结积分，审批通过后扣减，审批拒绝则解冻
- 团队价值报告默认按组织维度汇总，支持按项目维度切换；个人价值报告汇总到所属团队
- 价值积分永久累积，支持按时间段筛选，积分不清零
- 只提供数据参考，不直接驱动决策

### 4.5 关键状态机

#### 4.5.1 商机状态

```
发现 → 评估中 → 可行 ──→ MRD已通过
            │
            └─→ 不可行 ──→ 已归档
```

**MRD状态（详见4.3.1）：** 草稿 → 评审中 → 已通过/已拒绝 → 已归档

#### 4.5.2 项目状态

```
已创建 → 启动 → 进行中 → 交付中 → 收口 → 归档
            │
            ├─→ 已暂停
            └─→ 已取消
```

#### 4.5.3 任务状态

```
待分配 → 待开始 → 进行中 → 待审查 → 已完成 → 已验收
                │                          │
                ├─→ 已阻塞                 └─→ 待返工 → 进行中
                └─→ 已取消
```

#### 4.5.4 数字员工原型状态

```
设计中 → 测试中 → 已发布 ──→ 已下线
                │
                └─→ 已废弃
```

#### 4.5.5 技术资产状态

```
待审核 → 已收录 → 已发布 ──→ 已废弃
```

#### 4.5.6 知识条目状态

```
草稿 → 审核中 → 已发布 ──→ 已更新 → 已归档
```

**状态说明：**
- 已更新：内容有变更但仍在使用；知识条目内容变更后自动进入"已更新"状态，重大更新需重新审核（回到审核中），minor更新直接发布
- 已归档：知识已过时或被淘汰（触发条件：手动标记过时、新版本替代、长期未使用）

#### 4.5.7 风险项状态

```
已识别 → 评估中 → 处理中 → 已缓解 → 已关闭
                          │
                          └─→ 已接受
```

#### 4.5.8 数字员工实例状态

```
创建中 → 待分配 → 执行中 ──→ 闲置 ──→ 已回收
                │
                ├─→ 异常 ──→ 已销毁（重启失败）
                │
                └─→ 异常 ──→ 恢复成功 → 执行中
```

**状态说明：**
- 异常触发条件：Body运行故障（节点宕机、资源耗尽、脚本执行失败）
- 异常后可尝试重启恢复，恢复成功返回执行中，重启失败进入已销毁
- 闲置实例超过配置阈值后自动回收，或手动触发回收；闲置实例占用资源，不建议长期保持闲置
- "创建中"由数字员工域发起实例化请求时触发，基础设施域接收请求后分配资源、加载Body定义、初始化灵魂配置，完成后进入"待分配"

#### 4.5.9 任务规范状态

```
草稿 → 已发布 ──→ 已下线
```

**状态说明：**
- 任务规范版本变更需重新发布
- 已下线的任务规范不再用于新任务分配，已执行中的任务不受影响

#### 4.5.10 改进需求状态

```
待处理 → 处理中 → 已验证 → 已关闭
              │
              └─→ 已搁置
```

#### 4.5.11 价值规则状态

```
草稿 → 待审批 → 已启用
              │
              └─→ 已停用（新版本启用后旧版本自动停用）
```

**状态说明：**
- 价值规则支持多版本并存，只有一条规则处于"已启用"状态
- 规则变更时创建新版本，审批后切换启用状态，旧版本自动停用
