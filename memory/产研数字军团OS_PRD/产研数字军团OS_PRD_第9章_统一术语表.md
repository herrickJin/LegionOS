# 产研数字军团OS PRD

## 9. 统一术语表

| 术语 | 英文 | 定义 |
|---|---|---|
| **产研数字军团OS** | R&D Digital Legion OS | 产研数字军团操作系统，构建在技术资产、知识体系、项目库、业务系统之上，通过AI Agent调度资源、适配流程，为四大角色提供从商机洞察到交付运营的全流程产研能力支持 |
| **AI Native** | AI Native | AI能力被嵌入组织底层，成为业务设计、信息流转、流程编排和知识管理的基础环境；AI承担标准化执行、信息整合和流程推进，人的价值向判断力、创造力、边界设定、责任承担和关系经营迁移 |
| **数字员工** | Digital Employee | 在OS中承担某一角色、具备特定思维模式（workflow）、技能和执行工具的AI Agent，能承担和完成特定职责，与人类共享上下文，24小时无休执行标准化任务；人始终对最终交付结果承担责任。数字员工由灵魂和Body两部分组成 |
| **数字员工原型** | Digital Employee Prototype | 数字员工的设计模板，包含数字员工的灵魂和部署规格说明书，用于创建数字员工实例 |
| **数字员工实例** | Digital Employee Instance | 被分配到具体项目和任务中执行工作的数字员工，包含灵魂、Body以及具体项目和任务的Memory和上下文 |
| **数字员工灵魂** | Digital Employee Soul | 数字员工实例中的AI智能部分，包含思维模式（workflow）、知识库、技能配置和行为准则 |
| **数字员工Body** | Digital Employee Body | 数字员工的运行节点，负责承载灵魂、执行计算任务、与外部系统交互，可按负载动态扩缩容 |
| **经营管理者** | Business Manager | 对部门经营结果负责，从商机洞察到商业价值交付的全链路决策者 |
| **复合型研发经理** | Hybrid R&D Manager | 负责项目交付与团队效能，连接经营目标与技术执行的枢纽角色；需具备抽象业务能力、产品方案能力、AI工程化能力、架构设计能力、客户服务意识 |
| **AI工程师** | AI Engineer | 在OS驱动下，利用AI工具与数字员工协同完成具体研发任务，同时持续跟进前沿技术动态的执行角色 |
| **AI Agent架构师** | AI Agent Architect | 负责产研军团OS持续建设与技术资产运营，推动AI工程化实践落地，保障项目质量与团队效能的架构角色 |
| **商机** | Business Opportunity | 具有潜在商业价值的市场机会或客户需求，需经过可行性评估后决定是否推进 |
| **MRD** | Market Requirements Document | 市场需求文档，包含市场背景、需求分析、竞品对标、资源需求、预期收益 |
| **PRD** | Product Requirements Document | 产品需求文档，作为需求确认阶段的核心交付物，明确交付范围、功能边界和验收标准 |
| **交付承诺清单** | Delivery Commitment | 需求方与研发团队对齐后产出的正式交付约定，覆盖交付范围、需求分级、周期、资源、变更规则、风险与预案 |
| **里程碑** | Milestone | 基于任务分解的关键路径制定的阶段目标，包含完成标准、预计时间和责任人 |
| **关键路径** | Critical Path | 影响项目整体排期的任务序列，关键路径上的任务延期将直接导致项目延期 |
| **技术债务** | Technical Debt | 因时间压力或临时方案导致的代码质量问题，包括硬编码、跳过边界处理、TODO/FIXME/HACK等遗留项 |
| **技术资产** | Technical Asset | 可被人和数字员工复用的技术成果，包括公共组件、系统框架、工具脚本、自动化流水线、工具软件 |
| **知识体系** | Knowledge Base | 团队踩坑总结、方法论沉淀、规范文档、经验总结、工具使用指南、项目架构决策记录、团队问题与解答的集合，支持检索和持续更新 |
| **方案** | Solution / Playbook | 一个可以稳定、良好解决某一类问题的标准化方案，包括任务流规范、任务处理流程、最佳实践和验收标准 |
| **任务流规范** | Task Specification | OS的核心抽象，定义"什么人在什么条件下执行什么动作、产出什么、如何验收"。包含6要素：元数据、输入项与输入标准、产出项与验收标准、子任务规范清单与任务流、执行岗责任要求、执行工作台。项目规范、任务规范、子任务规范都是任务流规范的实例 |
| **任务流规范6要素** | Task Spec 6 Elements | 任务流规范的六个组成部分：①元数据（标识/名称/版本/类型）②输入项与输入标准 ③产出项与验收标准 ④子任务规范清单与子任务间任务流 ⑤执行岗责任要求（执行者类型/技能要求/判定规则/责任边界）⑥执行工作台（定制UI或规范说明页） |
| **项目规范** | Project Specification | 任务流规范的一种类型，顶层任务流，由MRD触发，包含多个任务规范 |
| **任务规范** | Task Specification | 任务流规范的一种类型，可独立执行的任务流，由项目规范或人工触发。**注：此术语已演进为"任务流规范"，保留此处供追溯** |
| **子任务规范** | Sub-task Specification | 任务流规范的一种类型，嵌套在任务规范内，不可独立触发，必须由父任务流节点触发 |
| **执行工作台** | Execution Workbench | 任务执行时的界面，可以是定制UI（如轻量IDE面板、Review面板）或规范说明页。OS内完成任务上下文展示和结果提交，实际执行可在OS外完成 |
| **典型任务规范（MVP）** | Typical Task Specs | MVP阶段预定义的**15个**任务规范：T1-MRD驱动的项目、T2-Old Java项目梳理、T3-Old Web项目梳理、T4-前端代码开发、T5-Java代码开发、T6-测试用例开发、T7-代码Review、T8-测试执行、T9-需求层设计、T10-架构层设计、T11-概要设计、T12-详细设计、T13-专项设计、T14-设计验证与交付、**T15-日志文件分析** |
| **项目驾驶舱** | Project Cockpit | T1任务规范的工作台，聚合展示MRD、里程碑看板、任务列表、进度追踪 |
| **代码分析面板** | Code Analysis Panel | T2/T3任务规范的工作台，可视化展示AI代码分析结果：结构图谱、依赖关系、问题清单、复杂度报表 |
| **轻量IDE面板** | Lightweight IDE Panel | T4/T5任务规范的工作台，提供代码浏览、组件库/API文档、实时预览、提交入口；编码在本地IDE完成 |
| **Review面板** | Review Panel | T7任务规范的工作台，提供Diff查看、行级批注、Approve/Reject、规范检查；全流程OS内完成 |
| **测试面板** | Test Panel | T6/T8任务规范的工作台，提供用例管理、执行触发、结果展示、缺陷上报 |
| **设计面板** | Design Panel | T9-T14任务规范的工作台，提供模板引导、画布编辑、AI辅助、组件库、版本对比；支持业务建模、架构设计、模块划分、类图、专项checklist、评审报告；全流程OS内完成 |
| **日志分析面板** | Log Analysis Panel | **T15任务规范的工作台，提供日志可视化、异常事件列表、根因分析、趋势报告；AI自动分析导入，人工确认** |
| **文档编辑器** | Document Editor | 已演进为设计面板的一部分，T9-T14设计文档在设计面板内编辑；原MRD文档编写功能由经营决策域独立提供 |
| **Prompt资产** | Prompt Asset | 经过验证和优化的提示词模板，用于驱动数字员工完成特定任务 |
| **数字员工性能仪表盘** | Agent Performance Dashboard | 展示数字员工响应成功率、任务完成率、用户采纳率、平均处理时长等核心指标的监控面板 |
| **经营驾驶舱** | Business Cockpit | 面向经营管理者的数据汇总视图，持续展示项目、资源、成本、质量和经营指标 |
| **个人工作台** | Personal Workbench | **SC-10场景的核心界面，聚合展示我的待办、我的任务、我的消息；点击任务打开对应工作台；只读聚合层，业务数据修改回源到对应域** |
| **MTTR** | Mean Time To Repair | 平均修复时间，从故障发生到恢复正常的时间间隔 |
| **Token消耗** | Token Consumption | AI模型调用过程中产生的资源消耗计量单位，用于成本核算和资源管控 |
| **AI工程化** | AI Engineering | 将AI能力系统化地融入研发流程的实践，包括工具选型、任务流设计、Prompt工程、效果评估和持续优化 |
| **人机共生** | Human-AI Symbiosis | AI辅助人提高认知效率与广度，人的认知深度与价值判断决定产品高度；技能不再是瓶颈，认知才是瓶颈 |
| **意图驱动** | Intent-Driven | 人想清楚、定义问题，数字员工探索解空间，人在价值岔路口做决策的研发范式 |
| **经验资产化** | Experience Assetization | 将个人隐性经验转化为组织显性资产的过程，确保知识不随人员流动而流失 |
| **系统接口化** | System API-First | 系统不仅给人使用，也给AI数字员工使用，让AI能够从业务层面推动产品发展 |
| **关键场景（SC）** | Key Scenario | **OS中用户可感知的端到端价值单元，跨域、可演示、有价值闭环。当前定义12个场景：SC-01任务流规范定义与执行、SC-02设计类任务执行、SC-03执行评价回流与规范优化、SC-04技术资产沉淀与复用、SC-05价值积分计算与兑换、SC-06改进需求跟踪与解决、SC-07跨组织项目资源调配、SC-08项目新人快速上手、SC-09项目进度监控与汇报、SC-10个人工作任务处理、SC-11外部系统任务执行、SC-12数字员工的定义与实例化** |
| **日志文件分析（T15）** | Log File Analysis | **MVP新增任务规范，数字员工自动分析日志文件，识别异常事件、定位根因、生成修复建议和趋势报告** |
| **外部系统任务执行** | External System Task Execution | **SC-11场景的核心能力，OS与外部系统（Ones、日志平台、测试平台）集成，触发外部任务并回写结果到OS** |
| **数字员工原型** | Digital Employee Prototype | **数字员工的设计模板，包含角色定位、技能矩阵、能力边界，用于创建数字员工实例** |
| **数字员工灵魂** | Digital Employee Soul | **数字员工实例中的AI智能部分，包含思维模式（workflow）、知识库（私有工作记忆）、行为准则、Prompt资产** |
| **数字员工Body** | Digital Employee Body | **数字员工的运行节点，负责承载灵魂、执行计算任务、与外部系统交互，可按负载动态扩缩容** |
| **数字员工实例** | Digital Employee Instance | **被分配到具体项目和任务中执行工作的数字员工，包含灵魂、Body以及具体项目和任务的Memory和上下文** |
| **数字员工管理面板** | Agent Management Panel | **SC-12场景的工作台，提供数字员工原型定义、灵魂配置、Body部署规格、实例化管理、性能仪表盘** |
