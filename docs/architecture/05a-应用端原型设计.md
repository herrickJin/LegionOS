## 1. 信息架构

### 1.1 导航结构

应用端采用**角色动态导航 + 流程驱动工作台**——用户登录后根据角色看到不同的侧边栏菜单，点击任务后进入流程驱动的专属工作台。**流程对用户不可见**，用户只看到"我的任务"列表，点击任务后工作台自动适配。

```

├── 我的任务（/tasks）— 任务列表，按类型筛选（BUG修复/故障处理/需求开发等）

│       ├── BUG修复工作台（/tasks/:taskId，flow=ones-bug）
```

### 1.2 路由规则

- 项目级页面统一携带 `projectId` 作为路径参数或查询参数
- 切换项目上下文时，带 `projectId` 的页面重新请求数据
- 不带 `projectId` 的页面（如个人工作台、消息中心）不受项目切换影响
- 消息跳转：消息条目携带目标路由和参数，点击后导航到对应页面

```
│          │ │          页面内容                      │  │
│          │ │                                      │  │

│          │ │                                      │  │
```

**四个区域**：

**顶部导航栏**（固定高度 56px）
- 左侧：Logo + 产品名称
- 中部：全局搜索（支持搜索项目、任务、需求、人员）
- 右侧：当前项目切换下拉 + 消息通知入口（带未读数 badge）+ 用户头像下拉（个人设置、退出登录）

**侧边栏**（宽度 220px，可收起至 64px）
- 顶部：当前角色标识
- 菜单项按角色视图分组显示
- 支持折叠子菜单
- 底部固定：帮助文档入口

**主内容区**
- 顶部面包屑导航
- 内容区域根据页面类型不同布局不同（列表页、详情页、看板页、看板+详情组合页）

**右侧面板**（按需出现，宽度 400px）
- 任务详情、消息详情等场景下以抽屉形式从右侧滑出
- 不使用时主内容区占满剩余宽度

### 2.2 流程驱动工作台

工作台是应用端的核心交互模式。**流程是隐式基础设施，不是用户的 UI 概念**——用户不会在界面上看到"流程"二字或流程编号，他们只看到任务。当用户点击一个任务时，工作台自动适配为该任务所属流程的专用布局。

#### 2.2.1 设计原则

1. **任务是入口，流程是底层**：侧边栏只有"我的任务"，没有"流程工作台"。用户从任务列表点击任务，工作台自动适配
2. **角色不决定布局，流程决定布局**：同一个研发经理，处理"BUG修复"和"故障处理"时的工作台形态完全不同
3. **流程对用户不可见**：任务卡片上显示的是任务类型标签（如"BUG修复""故障处理"），而非流程编号（F1/F2）。流程编号仅作为系统内部标识
4. **预定义 + 推断 + 回退**：核心流程预定义工作台模板，未知流程通过属性组合推断，无法匹配时回退到通用工作台

#### 2.2.2 工作台五区域布局

```
┌─────────────────────────────────────────────────┐
│  ① 任务上下文栏（Task Context Bar）                │
│  任务类型标签 · 任务标题 · 紧急度 · 当前步骤 · 操作按钮│
├──────────┬──────────────────────┬────────────────┤
│ ② 上下文  │  ③ 主操作区            │  ④ 数字员工区   │
│  面板     │  (Primary Action Zone) │  (AI Zone)     │
│  (可折叠)  │                       │  (可折叠)      │
│          │                       │               │
│  关联信息  │  核心工作内容            │  对话式交互     │
│  历史记录  │  随流程步骤变化          │  角色随流程变化  │
│  参考文档  │                       │               │
├──────────┴──────────────────────┴────────────────┤
│  ⑤ 流程进度条（Progress Tracker）                   │
│  ●──●──○──○──○  当前步骤高亮，可点击跳转             │
└─────────────────────────────────────────────────┘
```

**① 任务上下文栏**（固定顶部，高度 48px）
- 左侧：任务类型标签（带色彩编码，如"BUG修复""故障处理"）+ 任务标题 + 紧急度徽标
- 中部：当前步骤标识（如"第 2 步 / 共 5 步：定位代码"）
- 右侧：关键操作按钮（随步骤变化，如"确认复现"→"提交修复"→"标记已解决"）

**② 上下文面板**（可折叠，宽度 280px，默认展开）
- 内容随流程类型和当前步骤变化
- 展示与当前操作相关的支撑信息（BUG 详情、需求文档、变更记录、相关任务等）
- 支持搜索和快速导航

**③ 主操作区**（核心区域，占剩余宽度）
- 内容和布局完全由流程类型决定：代码编辑器、审批表单、监控面板、diff 视图等
- 随流程步骤切换，主操作区内容平滑过渡

**④ 数字员工区**（可折叠侧栏，宽度 320px，默认展开）
- 对话式交互界面
- 数字员工角色和预设提示语随流程类型变化
- 支持 SSE 流式输出预览

**⑤ 流程进度条**（底部固定，高度 36px）
- 显示当前流程的所有步骤，已完成步骤标记为实心，当前步骤高亮
- 可点击跳转到任意步骤（跳转后上下文面板和主操作区联动刷新）

#### 2.2.3 核心预定义流程

| 编号 | 流程名称（用户可见） | 触发来源 | 流程步骤 | 工作台特征 |
|------|---------|---------|---------|-----------|
| F1 | BUG修复 | ONES 工单指派 | 确认复现→定位代码→修复→自测→提交验证 | 左侧 BUG 详情+复现步骤，中间代码区，右侧 TestGen 对话 |
| F2 | 故障处理 | 告警/监控 | 止损降级→根因定位→热修复→部署→监控回归→复盘 | 顶部红色紧急栏，主区监控图表+日志流，数字员工实时分析 |
| F3 | 需求开发 | 迭代计划 | 技术设计→编码实现→单元测试→代码审查→合并 | 左侧需求文档+验收标准，中间代码区，右侧 CodeReview 对话 |
| F4 | 项目立项 | 人工发起 | 填写立项书→经营评审→资源分配→启动会 | 主区表单，左侧参考模板，右侧 DocWriter 辅助 |
| F5 | 代码审查 | PR/任务 | 浏览变更→逐文件审查→标注问题→确认通过 | 左侧文件树，中间 diff 视图，右侧审查清单+AI 建议 |
| F6 | 交付验收 | 里程碑触发 | 交付物检查→效果评估→客户确认→归档 | 主区交付物清单+声明vs证据，数字员工展示效能指标 |

#### 2.2.4 属性推断规则

当任务没有明确关联到预定义流程时，通过属性组合推断流程类型：

| 属性组合 | 推断流程 | 优先级 |
|---------|---------|--------|
| 来源=生产告警 AND 类型=BUG | F2 生产故障处理 | 高 |
| 优先级=P0 AND 来源=生产 | F2 生产故障处理（覆盖规则） | 最高 |
| 来源=ONES 工单 AND 类型=BUG | F1 ONES BUG 修复 | 高 |
| 来源=迭代计划 AND 类型=需求 | F3 需求开发 | 中 |
| 来源=ONES 工单 AND 类型=需求 | F3 需求开发 | 中 |
| 来源=PR 提交 AND 类型=审查 | F5 代码审查 | 中 |
| 来源=里程碑 AND 类型=验收 | F6 交付验收 | 中 |
| 无法匹配 | 通用工作台（可配置布局） | — |

#### 2.2.5 通用工作台（回退方案）

当任务流程无法匹配到任何预定义流程时，展示通用工作台：
- 上下文面板：任务基本信息（标题、描述、优先级、指派人）
- 主操作区：任务详情编辑 + 关联内容列表
- 数字员工区：通用助手（可自由对话）
- 进度条：简化为"待办→进行中→已完成"三态

#### 2.2.7 工作台原子组件规格

工作台采用**三层解耦架构**：框架层（固定布局）→ 组件层（10种可插拔原子组件）→ 配置层（流程×步骤→组件映射）。

##### 组件清单与跨流程复用

| # | 组件名 | 使用流程 | 核心特性 |
|---|--------|---------|---------|
| 1 | CodeEditor | F1/F2/F3 | 4种工作模式（定位/修复/测试/对比），流程中最常用的组件 |
| 2 | DiffView | F3/F5 | 独立跨文件diff，带文件树导航，区别于CodeEditor的内联对比 |
| 3 | MetricBoard | F2/F6 | 实时监控指标卡片（F2=时序折线图，F6=聚合进度条） |
| 4 | LogStream | F2 | 终端风格日志流，正则过滤，关键字高亮 |
| 5 | CheckList | F5/F6 | "声明vs证据"载体，三列结构（声明/证据/状态），不可删除 |
| 6 | RichEditor | F3/F4 | Markdown编辑器，模板系统，AI辅助插入 |
| 7 | FormEdit | F4 | 结构化表单，分区布局，草稿保存+版本对比 |
| 8 | InfoCard | F1/F5 | 只读信息卡片，类型：BUG详情/需求规格/任务规范/人员 |
| 9 | AIChat | 全部 | 对话式界面，数字员工身份随流程切换，SSE流式输出 |
| 10 | Timeline | F6 | 纵向时间线，节点类型：状态变更/评论/附件/系统事件 |

##### CodeEditor 详细规格

**4种工作模式**：

| 模式 | 使用场景 | 行为 | 侧边栏联动 |
|------|---------|------|-----------|
| 定位模式 | F1步骤2「定位代码」、F2步骤2「根因定位」 | 只读，AI标注可疑行，支持搜索跳转 | BUG复现步骤 / 错误堆栈 |
| 修复模式 | F1步骤3「修复」、F2步骤3「热修复」、F3步骤2「编码实现」 | 可编辑，语法高亮+补全，保存触发lint+构建验证 | 需求规格 / 修复方案 |
| 测试模式 | F1步骤4「自测」、F3步骤3「单元测试」 | 上半测试文件编辑器+下半执行输出，▶执行按钮 | AIChat切换到TestGen |
| 对比模式 | F3步骤4过渡 | 只读，同文件两版本内联对比 | 变更说明 |

**定位模式交互**：
- 打开自动跳转到AI推断可疑行（错误日志+堆栈匹配）
- 行号旁标记：AI标注=紫色圆点，人工标注=橙色圆点
- 点击标记→右栏AIChat弹出该行上下文分析
- 工具栏：搜索(⌘F) | 全文搜索(⌘⇧F) | 跳转行(⌘L) | 下一个标注(⌘→)
- 不可编辑，可添加行内评论（锚定行号）

**修复模式交互**：
- 标准编辑器行为（语法高亮、自动补全、错误提示）
- 保存(⌘S)触发：lint检查→构建验证→AI快速审查
- 右下角AI实时建议浮窗（类Copilot inline suggestion）
- 多文件Tab切换

**测试模式交互**：
- 上半=测试文件编辑器（可写）
- 下半=执行输出面板（终端风格）
- 顶部▶执行按钮
- 执行结果通过AIChat的TestGen代理解读

##### MetricBoard 详细规格

**F2 故障处理 — 实时监控仪表盘**（数据源：SSE推送，2-5秒刷新）：

| 指标卡片 | 展示形式 | 告警阈值 |
|---------|---------|---------|
| HTTP 5xx错误率 | 实时数字+60秒折线 | >5%红色闪烁 |
| P99响应时间(ms) | 实时数字+60秒折线 | >2000ms橙色 |
| QPS流量 | 实时数字+60秒折线 | 突降>50%红色 |
| CPU/内存/连接池 | 三行进度条 | >90%红色 |

交互：卡片全屏→15分钟大图；数值变化颜色脉冲；点击卡片→LogStream自动过滤该维度。

步骤联动：步骤1止损（占主区60%）→ 步骤2根因（缩到上半+下半LogStream）→ 步骤5监控回归（全屏+CheckList叠加）

**F6 交付验收 — 效能指标卡片**（数据源：服务端聚合计算）：

纵向列表，每个指标含：进度条+目标值+达标状态+对比变化。指标包括：任务按时完成率、数字员工初稿可用率、结果偏差信号次数、人工返工次数、需求变更次数。

关键区别：F2是时序数据关注"正在发生什么"，F6是聚合数据关注"周期做成什么"。F6每项标注目标值和达标状态，是"声明vs证据"的量化体现。

##### CheckList 详细规格 — "声明vs证据"的载体

**三列结构**：
1. **声明列**：文字描述可验证成果（"做到了什么"而非"做了什么"）
2. **证据列**：可点击链接/附件/截图/自动化报告
3. **状态列**：✓已验证 / ⚠待验证 / ✗有异议 / ○待提交

**F5 代码审查CheckList**：
- 粒度：代码级别（方法/变量/安全）
- 证据来源：AI扫描+人工审查，标注即可
- AIChat(CodeReview)自动生成草稿，每项附带扫描结果
- 审查者修正声明、补充证据、调整状态
- 证据类型标记：[自动扫描] [覆盖率报告] [人工审查] [链接]

**F6 交付验收CheckList**：
- 粒度：交付物级别（文档/测试/部署）
- 证据来源：自动化报告+人工提交，需可点击链接
- 每项需验证人签名+时间戳
- 验收标准：全部已验证=通过

**"声明vs证据"交互规则**：
1. 声明由谁写：AI代理可生成初始草稿，人类可修改
2. 证据由谁提交：自动化工具自动关联 / 人类手动上传
3. 谁来验证：系统自动验证（比对数值）/ 人类验证（指定验证人）
4. 异议处理：状态=有异议→自动生成讨论消息→关联到AIChat
5. 不可删除：CheckList项只能改状态不能删除（审计追踪）

##### 其他组件简要规格

**DiffView**：左侧变更文件树（+32/-8统计）+ 右侧diff区域（Unified/Side-by-side切换），差异块可添加评论锚定到行范围。F5代码审查步骤1-3专用。

**LogStream**：终端风格等宽字体深色背景，正则实时过滤，ERROR红/WARN橙/INFO白高亮，自动滚底+手动上翻暂停，右键复制/发送给AI分析。F2故障处理专用。

**RichEditor**：Markdown编辑器实时预览，支持嵌入任务链接/需求链接/@提及/图片，模板系统（立项书/技术方案/复盘报告），AI辅助（DocWriter可直接插入段落）。F3/F4共用。

**FormEdit**：结构化表单，字段类型（文本/数字/日期/下拉/多选/人员选择器/文件上传），分区布局，实时校验，草稿保存+版本对比。F4项目立项专用。

**InfoCard**：只读信息卡片，标签-值对列表+可折叠段落，支持嵌入代码片段/链接列表/人员头像。类型：BUG详情/需求规格/任务规范/人员。F1/F5共用。

**AIChat**：对话式界面，数字员工身份随流程切换（TestGen/IncidentBot/CodeReview/DocWriter/ProgressMonitor），预设提示语随步骤变化，SSE流式输出，支持文本/代码块/表格/链接/操作按钮。所有流程共用。

**Timeline**：纵向时间线从新到旧，节点类型（状态变更/评论/附件/系统事件），支持筛选只看人工/只看系统事件。F6交付验收的执行记录展示。

### 2.3 页面类型模板

应用端的页面分为四类基础布局模板，所有具体页面都在模板上填充内容：

**列表页模板**
- 顶部：页面标题 + 操作按钮（新建等）
- 筛选栏：搜索框 + 条件筛选器 + 排序
- 内容区：数据表格 / 卡片列表（支持分页或无限滚动）
- 批量操作栏（选中条目后底部浮现）

**详情页模板**
- 顶部：返回按钮 + 页面标题 + 操作按钮组
- 内容区：Tab 切换不同维度的信息
- 每个 Tab 内部按模块划分区域

**看板页模板**
- 顶部：页面标题 + 筛选器（按负责人/优先级/项目）
- 内容区：多列看板（列头为状态，卡片为任务）
- 支持拖拽卡片切换列

**看板+详情组合页模板**
- 左侧：看板（占 60% 宽度）
- 右侧：选中卡片的详情面板（占 40% 宽度）
- 点击不同卡片，右侧面板实时刷新

## 7. 数据流向总览

### 7.1 交互模式分类

应用端与 Server 层的交互按模式分为四类：

**模式 A：标准 REST 请求-响应**

页面加载、表单提交、CRUD 操作。绝大多数页面功能属于此类。

前端 → `GET/POST/PUT/DELETE /api/*` → Server → 返回 JSON

**模式 B：SSE 实时推送**

需要服务端主动推送信息的场景：

| 场景 | SSE 通道 | 触发条件 |
|------|----------|---------|
| 消息通知 | `/api/message/stream` | 有新系统通知、预警提醒、@消息 |
| 任务状态变更 | 由消息中心统一推送 | 任务被分配、状态变更、被验收 |
| 数字员工流式输出 | `/api/ai/collab/stream/:id` | 数字员工正在执行生成任务 |
| 风险预警 | 由消息中心统一推送 | 系统检测到延期/偏差信号 |
| 审批结果 | 由消息中心统一推送 | 审批通过/驳回 |

**模式 C：文件上传**

产出物提交、资产贡献、头像上传等场景。

前端 → `POST /api/*` (multipart/form-data) → Server → 文件存储服务 → 返回文件 URL

**模式 D：文件下载/导出**

报告导出、资产下载等场景。

前端 → `GET /api/*/export` → Server → 返回文件流（PDF/Excel）

### 7.2 实时推送架构

```
用户浏览器
    │
    ├── SSE 连接 1：/api/message/stream（所有推送消息统一通道）
    │       接收：任务变更、审批结果、风险预警、@消息、系统通知
    │
    ├── SSE 连接 2：/api/ai/collab/stream/:executionId（数字员工流式输出，按需建立）
    │       接收：数字员工生成内容（逐 chunk 推送）
    │
    └── 标准HTTP 请求（页面加载、表单提交等按需发起）
```

设计要点：

- 推送消息统一走一个 SSE 通道，前端根据消息类型分发到不同处理器
- 数字员工流式输出是独立 SSE 通道，只在查看执行结果时建立连接，执行完成或离开页面时断开
- 不使用 WebSocket（WebSocket 通道留给数字员工节点之间的内部通信）
