# T16 需求结构化与准入规范

> **定位**：把"一句话需求"转化为"可以进入设计阶段的结构化需求"，是 T9 需求层设计的前置步骤
> **对应一级规范**：06-一级产研流程规范 §2.2
> **状态**：初稿

---

## 0. 先看一个例子

**输入——复合型研发经理收到的需求：**

> 产品方在微信发了三条消息：
>
> （1）"我们的后台订单列表要加一个批量导出功能，供应商那边催了好几次了"
>
> （2）"导出格式就 Excel 吧，他们习惯用"
>
> （3）"对了导出的字段跟列表展示的差不多就行，加上供应商联系方式"

**输出——经过本规范处理后，产出的结构化需求文档：**

- **需求标题**：后台订单列表批量导出 Excel
- **一句话描述**：运营人员可以在订单列表页选中多笔订单，一键导出为包含供应商联系信息的 Excel 文件
- **涉及角色**：运营人员（导出）、供应商（接收文件）
- **不可协商条件**（1条）：必须支持大于 5000 条订单的导出
- **可讨论条件**（2条）：导出字段是否与列表完全一致 / 是否支持自定义选择字段
- **功能边界**
  - 包含：订单数据的筛选（按状态/日期/供应商）→ 勾选 → 导出 Excel → 文件下载
  - 不包含：定时自动导出 / 导出 PDF / 供应商端自助查询
- **关键业务规则**
  - 单次导出上限 10,000 条（超过提示分批导出）
  - 导出内容包含：订单编号、供应商名称、联系人、联系电话、订单金额、下单时间、状态
  - 导出文件名格式：`订单导出_20240604_1530.xlsx`
- **异常流程**（3个）：导出中页面关闭怎么办 / 导出失败怎么提示 / 无数据时怎么处理
- **验收标准**
  - 选中 100 条订单 → 点导出 → 5 秒内下载 Excel 文件
  - Excel 文件包含 7 个字段，字段名与列表一致
  - 无数据时点导出 → 提示"当前筛选条件下无订单数据"
- **待澄清问题**（3个）：导出历史是否需要保存？是否需要导出操作日志？供应商联系方式是否包含手机号？
- **准入结论**：✅ 通过，可进入 T9 需求层设计

## 3. 子任务规范 + 任务流

```
步骤1 → 步骤2 → 步骤3 → 步骤4 → 步骤5
 提取     歧义     补充     生成     确认
关键信息  消除     异常流程  结构化   与准入
(数字)   (数字)   (数字)    (数字)   (人)
```

### 步骤 1：提取关键信息 ⚙️ 数字员工

**输入**：原始需求内容

**数字员工做的事**：
1. 从原始需求中提取以下结构化字段，填空：
   - 需求标题（一句话总结）
   - 一句话描述
   - 涉及的用户角色
   - 需求来源（产品方诉求 / 客户反馈 / 线上问题 / 技术债务 / 合规要求）
2. 输出格式：每条提取结果后面标注"原文依据：xxx"，方便人核对

**产出物**：`[草案] 关键信息提取`（JSON 格式）

**判定规则**：该步骤由数字员工执行（技能成熟度 L2 — 文本提取与总结）

---

### 步骤 2：歧义消除 ⚙️ 数字员工

**输入**：步骤 1 产出 + 原始需求内容

**数字员工做的事**：
1. 扫描需求中可能产生歧义的地方，按三类逐一检查：
   - **模糊表述**：例如"差不多就行"、"跟以前一样"、"尽快"、"优化一下"
   - **缺失信息**：例如没说导出格式、没说权限、没说数据量级
   - **隐含假设**：例如"用户肯定知道怎么操作"、"这个字段肯定有"
2. 对每一个歧义点生成一个**追问问题**，问题需满足：封闭式优于开放式（"导出 Excel 还是 CSV？"优于"导成什么格式？"）
3. 标注每个问题的紧急程度：🔴 阻塞设计 / 🟡 影响范围 / 🟢 锦上添花

**产出物**：`[草案] 歧义清单`（每行一条：歧义原文 + 追问问题 + 紧急程度）

**判定规则**：该步骤由数字员工执行（技能成熟度 L2 — 文本分析与提问）

---

### 步骤 3：补充异常流程 ⚙️ 数字员工

**输入**：步骤 1 产出 + 步骤 2 产出

**数字员工做的事**：
1. 强制检查以下 5 类异常场景，为每一类生成至少 1 条异常流程描述：
   - **边界条件**：数据为空 / 数据量极大 / 输入格式错误
   - **权限边界**：无权限的人能否看到/操作
   - **并发冲突**：多人同时操作同一数据
   - **中断恢复**：操作进行中页面关闭/网络断开
   - **状态冲突**：操作时数据状态已变化（如订单已被别人导出/取消）
2. 每条异常流程包括：触发条件 + 预期行为
3. 标注哪些异常需要在 T9 设计阶段出交互稿（🔴=必须 / 🟡=建议）

**产出物**：`[草案] 异常流程清单`

**判定规则**：该步骤由数字员工执行（技能成熟度 L2 — 场景枚举与边界分析）

---

### 步骤 4：生成结构化需求 + 准入评估 ⚙️ 数字员工

**输入**：步骤 1/2/3 的全部产出

**数字员工做的事**：
1. 按以下模板生成结构化需求文档：
   - 需求标题
   - 一句话描述
   - 涉及角色
   - 不可协商条件 vs 可讨论条件
   - 功能边界（包含 / 不包含）
   - 关键业务规则
   - 异常流程
   - 验收标准（初步）
   - 待澄清问题（汇总步骤 2 的 🔴 和 🟡 问题）
2. 对需求进行准入评估，按以下 checklist 逐项判定：

| 准入检查项 | 通过标准 | 不通过时的动作 |
|-----------|---------|--------------|
| 功能边界是否清晰 | 至少有 2 条"包含"和 1 条"不包含" | 🔴 退回步骤 2，标记缺失项 |
| 至少 1 个用户角色 | 明确命名，非"用户"这种泛指 | 🟡 标"待确认"，不阻塞准入 |
| 至少 1 条验收标准 | 可量化（时间/数量/准确率） | 🔴 退回步骤 3，需数字员工补充 |
| 至少 2 条异常流程 | 覆盖边界条件和中断恢复 | 🟡 不阻塞，但标注"风险：异常流程不足" |
| 需求来源可追溯 | 知道是谁提的、什么时候提的 | 🔴 退回步骤 1，需人工补充 |
| 是否有不可协商条件 | 至少 1 条 | 🟡 允许没有（说明需求方未明确约束） |
| 是否关联到项目 | 关联到项目管理域的项目 ID | 🔴 退回，项目管理域未创建项目则无法准入 |

3. 输出准入结论：
   - ✅ 通过：所有 🔴 项通过 → 可进入 T9 设计
   - ⚠️ 条件通过：🔴 项通过但 🟡 项有缺口 → 可进入 T9，但标注风险项
   - ❌ 不通过：有 🔴 项未通过 → 返回步骤 2/3 补充，或退回需求方

**产出物**：`[草案] 结构化需求文档 + 准入评估结果`

**判定规则**：该步骤由数字员工执行（技能成熟度 L3 — 结构化生成 + 规则判定）

---

### 步骤 5：确认与修订 👤 复合型研发经理

**输入**：步骤 4 的完整产出

**人做的事**：
1. 逐项审查结构化需求文档，重点检查：
   - 数字员工的理解是否与原始需求一致（对照"原文依据"）
   - 不可协商条件是否准确（跟需求方确认过的才算）
   - 待澄清问题列表是否完整、优先级是否合理
2. 对歧义清单中的 🔴 问题，决定：
   - 自己可以回答 → 直接修改结构化文档
   - 需要跟需求方确认 → 标记为"待需求方回复"，不阻塞准入但标风险
3. 最终做出准入决策（审批通过 / 退回补充 / 退回需求方）
4. 审批通过后，结构化需求文档归档，自动触发 T9 任务创建

**产出物**：`[终稿] 结构化需求文档`（状态：已准入）

**判定规则**：该节点为人节点（审核节点），必须由复合型研发经理审批。审核不通过 → 驳回至步骤 4，附带修改意见。

## 4. 产出项与验收标准

### 4.1 产出物

| 产出物 | 格式 | 存放位置 |
|--------|------|---------|
| 结构化需求文档 | JSON（API） + Markdown（可读） | 项目管理域，关联项目 ID |
| 歧义清单 | 结构化列表 | 结构化需求文档的"待澄清问题"字段 |
| 准入评估结果 | 准入状态（通过/条件通过/不通过） + checklist 明细 | 结构化需求文档的"准入结论"字段 |
| 原文依据索引 | 需求原文片段 → 结构化条目的映射 | 结构化需求文档的"依据"字段 |

### 4.2 验收标准

**对结构化需求文档的质量要求：**

- [ ] 功能边界有至少 2 条"包含"和 1 条"不包含"，"不包含"需具体到功能点而非泛泛的"超出范围"
- [ ] 异常流程覆盖了全部 5 个强制检查维度（边界/权限/并发/中断/状态冲突）
- [ ] 每条验收标准可量化——有具体数字（时间/数量/准确率），而非"体验好""速度快"这种主观表述
- [ ] 待澄清问题均为封闭式提问（"A 还是 B？"而非"怎么做？"）
- [ ] 复合型研发经理已确认通过

**对准入流程的质量要求：**

- [ ] 准入 checklist 逐项完成，不得跳过
- [ ] 🔴 不通过项有明确的补充指引（"缺什么/去哪补/谁来补"），而非仅仅"不通过"
- [ ] 从原始需求输入到准入结论输出，全流程可追溯——每一步的依据（步骤 1 的"原文依据"→ 步骤 4 的"准入结论"）形成完整链路

## 5. 执行岗责任要求

| 角色 | 执行节点 | 技能要求 | 做什么 |
|------|---------|---------|--------|
| 数字员工 | 步骤 1-4 | 文本提取与总结（L2）、场景枚举与边界分析（L2）、结构化生成与规则判定（L3） | 全自动执行：提取关键信息 → 识别歧义 → 补充异常流程 → 生成结构化文档 + 准入评估 |
| 复合型研发经理 | 步骤 5（审核节点） | 对需求的业务判断力、对需求方的沟通能力 | 审查数字员工产出、回答可自行判定的问题、联系需求方澄清 🔴 问题、做出准入决策 |

**关键原则：**

- 步骤 1-4 完全由数字员工执行，人类只在步骤 5 介入
- 数字员工**不直接联系需求方**——需要需求方确认的问题，由数字员工生成追问清单，复合型研发经理选择：自己回答 / 转给需求方
- 准入决策权在人——数字员工只给"建议通过/建议退回"的评估，不做最终判断

### 6.2 工作台说明

- **左面板（40%）**：原始需求（可滚动，支持文本/图片/语音转录）+ 数字员工的分析对话，用户可查看每步的推理过程
- **右面板（60%）**：步骤 4 产出的结构化需求文档，实时可编辑——复合型研发经理可以直接修改任何字段，修改记录自动标注"人工修改"
- **右侧底部**：准入评估 checklist（每个检查项带状态灯：✅🟡🔴）+ 准入结论 + 三个操作按钮
- **底部进度条**：5 步进度，当前步骤高亮，已完成步骤可点击回溯（步骤 1-4 回溯会触发重新生成）

                   └── T9.3 交互原型设计
                         │

                   T10-T14 架构/详细/专项设计
