## 1. 领域定位与分类

产研数字军团OS 划分为 **12个业务域**，按战略重要性和依赖关系分为四类：

| 分类          | 域        | 战略价值 | 对AI Native的核心贡献          |
| ----------- | -------- | ---- | ------------------------ |
| **核心域（7个）** | 经营决策域    | 高    | 商机洞察→立项决策，AI辅助可行性评估      |
|             | 项目管理域    | 高    | 项目全生命周期管理，资源调配           |
|             | 项目实施域    | 高    | 任务执行与跟踪，人机协同的执行层         |
|             | 产研流程域    | 高    | **OS核心抽象**，定义流程规范与人机协同规则 |
|             | 数字员工域    | 高    | AI Agent的定义、实例化与行为管理     |
|             | 技术资产域    | 高    | 可复用技术成果的管理与推荐            |
|             | 知识经验域    | 高    | 组织知识沉淀与RAG检索支持           |
| **支撑域（2个）** | 用户权限域    | 中    | 统一身份体系，数字员工也是用户          |
|             | 基础设施域    | 中    | 数字员工Body运行时与项目上下文承载      |
| **辅助域（2个）** | 自我进化域    | 低    | 系统改进需求跟踪与优化闭环            |
|             | 价值评估与兑现域 | 低    | 人/数字员工/资产的价值量化与积分        |
| **视图层（1个）** | 个人域      | -    | 跨域数据聚合，个性化工作台            |

**分类依据（Eric Evans DDD）：**

- **核心域**：业务差异化所在，决定产品核心竞争力。产研流程域是OS区别于传统项目管理系统的关键——它定义"人和数字员工如何协同工作"。
- **支撑域**：业务需要但不决定差异化，需稳定运行。用户权限域采用类EHR设计，基础设施域聚焦运行时管理。
- **辅助域**：增强体验但不影响核心业务。价值评估域提供量化反馈，驱动自我进化。
- **视图层**：只读聚合层，不构成独立业务逻辑。

> **设计决策**：MVP阶段将4个高频联动核心域（经营决策、项目管理、项目实施、产研流程）聚合到 `pdm-svc`，降低分布式事务复杂度。

## 2. 上下文映射

域间关系采用DDD上下文映射模式，明确集成契约与依赖方向。

### 映射关系总览

```
┌────────────────────────────────────────────────────────────────────────┐
│                    产研数字军团OS — 上下文映射                          │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│  Customer-Supplier（客户-供应商）                                      │
│  ├── 经营决策域 → 项目管理域：MRD移交（Supplier：经营决策，Customer：项目）│
│  ├── 项目管理域 → 项目实施域：任务分解（Supplier：项目，Customer：实施）  │
│  ├── 产研流程域 → 项目实施域：任务规范驱动分配                          │
│  └── 技术资产域 → 项目实施域：资产推荐与复用                          │
│                                                                        │
│  Published Language（发布语言）                                        │
│  ├── 用户权限域 → 所有域：用户/角色/权限API（统一身份语言）             │
│  ├── 产研流程域 → 数字员工域：任务规范定义（工作流DSL）                │
│  └── 知识经验域 → 产研流程域：任务FAQ（知识检索API）                  │
│                                                                        │
│  Open Host Service（开放主机服务）                                      │
│  ├── 基础设施域 → 数字员工域：Body运行时API（Restful）                  │
│  └── 价值评估域 → 所有域：价值积分计算API                            │
│                                                                        │
│  Conformist（跟随者）                                                 │
│  ├── 自我进化域 → 所有域：改进需求订阅（被动接收事件）                  │
│  └── 个人域 → 所有域：只读聚合（不修改源数据）                        │
│                                                                        │
│  Anti-Corruption Layer（防腐层）                                       │
│  ├── 数字员工域 → 外部LLM API：模型路由ACL                            │
│  └── 基础设施域 → 外部GitLab/Jira：外部系统ACL                        │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

### 关键关系详解

**1. 经营决策域 → 项目管理域（Customer-Supplier）**
- **Supplier**：经营决策域，产出MRD
- **Customer**：项目管理域，消费MRD创建项目
- **契约**：`MRD.Approved`事件（RabbitMQ），包含MRD文档JSON
- **设计理由**：经营管理者审批后，项目由复合型研发经理接收并创建，确保经营决策与项目执行分离

**2. 产研流程域 → 项目实施域（Customer-Supplier）**
- **Supplier**：产研流程域，定义任务规范与判定规则
- **Customer**：项目实施域，按规范分配任务实例
- **契约**：`TaskSpec.Assigned`（任务规范绑定），`Node.Completed`（节点完成事件）
- **设计理由**：任务"如何执行"由产研流程域定义，"谁执行、执行状态"由项目实施域管理

**3. 用户权限域 → 所有域（Published Language）**
- **统一语言**：用户ID、角色、权限的JSON Schema
- **所有域**调用`iam-svc`验证用户身份与功能权限
- **数字员工**也是用户，功能权限由`iam-svc`管理，业务能力由数字员工域配置

**4. 数字员工域 → 外部LLM API（Anti-Corruption Layer）**
- **ACL位置**：`agent-svc`内部`LlmRouter`组件
- **职责**：隔离外部LLM API差异（OpenAI/Claude/通义千问），提供统一`generate()`接口
- **好处**：切换模型供应商不需要改业务代码

**5. 基础设施域 → 外部系统（Anti-Corruption Layer）**
- **ACL位置**：`inf-svc`内部`ExternalAdapter`组件
- **职责**：适配GitLab（代码仓库）、Jira（问题跟踪）、企业微信（消息推送）
- **配置化**：由AI Agent架构师在OS内配置适配器参数，无需改代码

### 映射模式选择依据

| 关系特征 | 选择模式 | 示例 |
|---------|----------|------|
| 上游团队掌控下游需求 | Customer-Supplier | 经营决策→项目管理 |
| 需要暴露API给多个域 | Published Language | 用户权限域 |
| 需要提供REST API给特定域 | Open Host Service | 基础设施域→数字员工域 |
| 下游只能被动接收上游数据 | Conformist | 自我进化域 |
| 外部系统集成 | Anti-Corruption Layer | LLM API、GitLab |

## 3. 核心聚合根与领域事件

### 聚合根识别（Aggregate Root）

每个域的聚合根是其内部实体的唯一入口点，外部只能通过聚合根访问内部对象。

| 域 | 聚合根 | 唯一标识 | 关键职责 |
|-----|---------|----------|----------|
| **经营决策域** | Opportunity（商机） | opportunity_id | 商机洞察、评估、立项决策，MRD是商机的产出物 |
| **项目管理域** | Project（项目） | project_id | 项目全生命周期管理，里程碑与资源协调 |
| **项目实施域** | Task（任务） | task_id | 任务实例生命周期，执行状态跟踪 |
|  | Resource（资源） | resource_id | 人/数字员工/设备的资源池管理，技能与可用时间 |
| **产研流程域** | TaskType（任务类型） | type_id | 任务分类与特征定义，决定适用场景 |
|  | Workflow（工作流） | workflow_id | 任务执行流程定义，节点与判定规则 |
| **数字员工域** | AgentPrototype（数字员工原型） | prototype_id | 定义数字员工的能力边界与灵魂配置 |
|  | AgentInstance（数字员工实例） | instance_id | 运行中的数字员工实例，绑定具体项目/任务 |
|  | RuntimeConfig（运行环境配置） | config_id | 数字员工运行的模型、工具、权限配置 |
| **用户权限域** | User（用户） | user_id | 统一身份，人/数字员工都是用户 |
|  | Role（角色） | role_id | 角色定义与权限集合 |
|  | Permission（权限） | permission_id | 功能权限与数据权限的原子定义 |
| **技术资产域** | TechAsset（技术资产） | asset_id | 可复用技术成果的管理与版本控制 |
| **知识经验域** | KnowledgeSystem（知识体系） | system_id | 知识分类体系，包含多个知识条目 |
|  | KnowledgeEntry（知识条目） | entry_id | 组织知识沉淀与检索 |
| **基础设施域** | DigitalEmployeeBody（数字员工Body） | body_id | 数字员工运行时容器，模型与工具加载 |
|  | Model（模型） | model_id | AI模型管理，版本与配置 |
|  | ProjectContext（项目上下文） | context_id | 项目运行时的上下文环境 |
|  | RuntimeNode（运行节点） | node_id | 数字员工Body运行环境与资源隔离 |
| **自我进化域** | ImprovementItem（可改进项） | item_id | 系统级改进跟踪，来源与优先级 |
| **价值评估与兑现域** | ValueItem（价值项） | item_id | 人/数字员工/资产的价值量化记录 |
|  | ValueRule（价值规则） | rule_id | 价值计算规则，积分策略配置 |

**聚合根设计原则：**
- 聚合根之间只能通过ID引用，不能直接对象引用（避免对象遍历导致级联加载）
- 聚合根内部保证业务规则一致性（如Task的状态流转规则在Task聚合内完成）
- 跨聚合操作通过领域事件实现最终一致性

### 关键领域事件（Domain Event）

领域事件捕获"业务中发生的重要事情"，用于跨域通信与审计。

**事件命名规范：** `Aggregate.EventName`（过去时态）

#### 经营决策域事件
| 事件 | 触发条件 | 负载（关键字段） | 订阅者 |
|------|---------|----------------|---------|
| `Opportunity.Created` | 经营管理者创建商机草稿 | opportunity_id, title, creator_id, status | 无（初期） |
| `Opportunity.Evaluated` | 商机评估完成 | opportunity_id, evaluation_result, recommendation | 经营决策域（内部流程） |
| `Opportunity.Converted` | 商机转化为MRD并审批通过 | opportunity_id, mrd_id, project_brief | **项目管理域**（创建项目） |
| `Opportunity.Rejected` | 商机被拒绝或关闭 | opportunity_id, rejected_at, reason | 无 |

#### 项目管理域事件
| 事件 | 触发条件 | 负载 | 订阅者 |
|------|---------|------|---------|
| `Project.Created` | MRD审批通过或OS外直接创建 | project_id, name, manager_id, source_mrd_id | 产研流程域、Portal-svc |
| `Project.MilestoneReached` | 里程碑所有关联任务完成 | project_id, milestone_id, reached_at | 经营决策域（汇报） |
| `Project.RiskIdentified` | 风险项升级 | project_id, risk_id, severity, description | 经营决策域、复合型研发经理 |

#### 产研流程域事件
| 事件 | 触发条件 | 负载 | 订阅者 |
|------|---------|------|---------|
| `TaskSpec.Published` | 任务规范发布新版本 | spec_id, version, applicable_scenarios | 项目实施域 |
| `Node.Completed` | 任务流节点执行完成 | task_id, node_id, result, completed_by（人/数字员工） | 项目实施域（触发下一节点） |

#### 项目实施域事件
| 事件 | 触发条件 | 负载 | 订阅者 |
|------|---------|------|---------|
| `Task.Assigned` | 任务分配给执行者 | task_id, assignee_id, assignee_type（HUMAN/AGENT） | 数字员工域（如分配给Agent） |
| `Task.Completed` | 任务所有节点完成并通过验收 | task_id, completed_at, deliverables | 项目管理域（更新进度）、价值评估域（计算积分） |
| `Task.Blocked` | 任务执行遇到阻塞项 | task_id, blocker_description, reported_by | 项目管理域（升级风险） |
| `Resource.AvailabilityChanged` | 资源可用时间发生变化 | resource_id, resource_type, new_availability, reason | 项目管理域（重新分配任务） |

#### 数字员工域事件
| 事件 | 触发条件 | 负载 | 订阅者 |
|------|---------|------|---------|
| `Agent.InstanceCreated` | 数字员工实例化 | instance_id, prototype_id, project_id, assigned_tasks | 基础设施域（分配运行资源） |
| `Agent.TaskAssigned` | Server推送任务给数字员工 | instance_id, task_id, task_spec, context | 数字员工节点（WebSocket推送） |
| `Agent.TaskCompleted` | 数字员工上报任务完成 | instance_id, task_id, result, evidence（执行日志/产出物） | 项目实施域（标记节点完成） |
| `Agent.PerformanceRecorded` | 数字员工执行结果评价 | instance_id, task_id, rating, feedback | 价值评估域（计算积分） |

#### 技术资产域事件
| 事件 | 触发条件 | 负载 | 订阅者 |
|------|---------|------|---------|
| `TechAsset.Published` | 技术资产审核通过并发布 | asset_id, name, type, version, contributor_id | 产研流程域（更新任务规范引用） |
| `TechAsset.Reused` | 技术资产被项目中任务复用 | asset_id, reused_by_task_id, reused_by_user_id | **价值评估域**（给贡献者计算积分） |

#### 知识经验域事件
| 事件 | 触发条件 | 负载 | 订阅者 |
|------|---------|------|---------|
| `KnowledgeEntry.Published` | 知识条目审核通过并发布 | entry_id, title, category, contributor_id | 产研流程域（关联任务FAQ） |
| `KnowledgeEntry.Cited` | 知识条目被任务执行引用 | entry_id, cited_by_task_id, cited_by_user_id | **价值评估域**（给贡献者计算积分） |

#### 价值评估与兑现域事件
| 事件 | 触发条件 | 负载 | 订阅者 |
|------|---------|------|---------|
| `ValuePoint.Calculated` | 积分计算完成 | target_id（人/数字员工/资产）, point_type, amount, reason | Portal-svc（个人工作台展示） |
| `ValuePoint.Redeemed` | 积分兑换申请审批通过 | user_id, point_amount, reward_type, approved_by | 经营决策域（记录兑现记录） |

### 事件流转时序示例

以"数字员工执行任务"为例，展示跨域事件流转：

```
经营决策域               项目管理域              产研流程域            项目实施域            数字员工域
    │                      │                     │                    │                  │
    │── MRD.Approved ──→│                     │                    │                  │
    │                      │── Project.Created ─→│                    │                  │
    │                      │                     │── TaskSpec.Loaded ─→│                  │
    │                      │                     │                    │── Task.Assigned ─→│
    │                      │                     │                    │                  │── Agent.TaskAssigned ─→
    │                      │                     │                    │                  │    （WebSocket推送）
    │                      │                     │                    │                  │←── Agent.TaskCompleted ─
    │                      │                     │←── Node.Completed ──│                  │
    │                      │←── Task.Completed ─│                    │                  │
    │                      │                     │                    │── ValuePoint.Calculated ─→
    │                      │                     │                    │  （异步，事件总线）
```

> **设计决策**：MVP阶段使用RabbitMQ实现事件总线，保障至少一次投递（at-least-once）。成长期引入Saga模式处理跨服务长事务。

## 4. 域→服务映射（MVP）

> **定位**：把 12 域 × 18 聚合根落到"4+1 服务"的物理部署单元
> **原则**：MVP 阶段"4+1 服务"是基于"高频联动 + 部署成本"的最优聚合，不代表长期形态
> **演进触发**：任一服务内聚合根数量 > 4、代码量 > 5 万行、团队 > 5 人，即触发拆分
> **依赖文档**：§1.5《架构全景与设计原则》核心概念对照表（术语统一）

### 4.1 4+1 服务总览

| 服务 | 装载域 | 装载的聚合根（MVP） | 数据库 | 拆分触发条件 |
|------|--------|-------------------|--------|--------------|
| **pdm-svc** | 经营决策域、项目管理域、项目实施域、产研流程域 | Opportunity, Project, Task, Resource, TaskType, Workflow | `core_db`（4 Schema 隔离） | 团队 > 5 人 or 域边界争议 > 2 次/月 |
| **asset-svc** | 技术资产域、知识经验域 | TechAsset, KnowledgeEntry（**MVP 不建模 KnowledgeSystem**） | `asset_db` | 知识检索 RPS > 100（独立为 knowledge-svc） |
| **agent-svc** | 数字员工域、基础设施域（部分） | AgentPrototype, AgentInstance, RuntimeConfig, RuntimeNode, Model, ProjectContext | `agent_db` | 数字员工实例 > 50（独立 infra-svc） |
| **portal-svc** | 个人域（视图层） | **无聚合根**，BFF 层 + 用户偏好 | `portal_db`（缓存 + 偏好） | QPS > 500（独立部署） |
| **iam-svc** | 用户权限域 | User, Role, Permission | `iam_db` | 长期不拆分 |

**MVP 聚合根总计**：6 + 2 + 6 + 0 + 3 = **17 个**（从原 18 减 1：去掉 KnowledgeSystem，扁平化处理）

### 4.2 MVP 域/聚合根的取舍决策

| 域 | MVP 范围 | MVP 后置 | 决策理由 |
|---|---|---|---|
| 经营决策域 | ✅ 全部 | — | 商业闭环必需 |
| 项目管理域 | ✅ 全部 | — | 商业闭环必需 |
| 项目实施域 | ✅ 全部 | — | 商业闭环必需 |
| 产研流程域 | ✅ TaskType + Workflow 基础版 | 高级流程编排引擎、Saga | 高级特性放 v2 |
| 数字员工域 | ✅ Prototype + Instance + RuntimeConfig | Agent 自学习、记忆机制 | 学习机制需要数据积累 |
| 技术资产域 | ✅ TechAsset CRUD + 检索 | 自动推荐、版本对比、依赖分析 | 推荐算法需要数据 |
| 知识经验域 | ⚠️ 只做 KnowledgeEntry（扁平存储，不分体系） | KnowledgeSystem（分类层级） | MVP 阶段扁平即可 |
| 用户权限域 | ✅ 全部 | — | 基础必需 |
| 基础设施域 | ⚠️ RuntimeNode + Model + ProjectContext；**DigitalEmployeeBody 简化为 K8s ConfigMap，不进数据库** | DigitalEmployeeBody 完整建模（v2 引入镜像版本管理） | MVP 用 ConfigMap 表达即可 |
| 自我进化域 | ⚠️ 只做埋点（收集改进需求事件），不做处理 | 改进项处理流程、自动分析、根因聚类 | 自我进化是 v2+ 能力 |
| 价值评估与兑现域 | ⚠️ 只做基础积分记录（完成任务 +N 分、被复用 +M 分），不做规则引擎 | 价值规则引擎、积分兑换、排行榜 | MVP 阶段硬编码规则足够 |
| 个人域（视图层） | ✅ 只读聚合 + 个人偏好（布局/通知） | — | 基础必需 |

### 4.4 数据库设计原则

- **独立库隔离**：`core_db / asset_db / agent_db / portal_db / iam_db` 共 **5 个独立数据库**，禁止跨库 JOIN
- **跨域查询**：必须经过 API 调用，不允许在应用层跨库查表（包括 PostgreSQL Foreign Data Wrapper 跨库访问也禁止）
- **portal_db 特殊性**：只包含用户偏好（布局、通知设置）+ 物化视图缓存（可重建），**无业务数据**
- **core_db 内部多 Schema**：MVP 阶段 4 个核心域（经营决策/项目管理/项目实施/产研流程）用 `opp_schema / proj_schema / task_schema / workflow_schema` 隔离
  - 跨 Schema 访问走同库不同 Schema 的**视图（VIEW）**，由对应域 owner 授权
  - 写操作**禁止**跨 Schema（必须走服务 API）

### 4.5 服务演进路线

| 阶段 | 时间窗 | 服务划分 | 拆分触发条件 |
|------|--------|----------|--------------|
| **MVP** | 0-4 周 | 4+1 服务（本节） | — |
| **成长 1** | 4-12 周 | 6 服务：从 pdm-svc 拆出 `workflow-svc`（产研流程独立） | workflow 复杂度上升 / 流程编排需求 |
| **成长 2** | 12-24 周 | 8 服务：从 agent-svc 拆出 `infra-svc`（基础设施独立） | 数字员工实例 > 50 / 资源调度需求 |
| **成熟期** | 24 周+ | 12 服务：每域独立 | 团队规模 > 30 人 / 业务复杂度过高 |

> **设计验证（每次拆分前）**：
> 1. 待拆分服务内聚合根数量 > 4？
> 2. 代码量 > 5 万行？
> 3. 团队维护人数 > 5？
> 4. 跨域调用频次（月均）> 1000 次？
> 5. **同时满足 ≥ 3 条**才触发拆分，避免过早微服务化

**图例说明：**
- **CS** = Customer-Supplier（客户-供应商）
- **OHS** = Open Host Service（开放主机服务）
- **PL** = Published Language（发布语言）
- **CF** = Conformist（跟随者）
- **ACL** = Anti-Corruption Layer（防腐层）

**关键映射关系解读：**

**1. 经营决策域 → 项目管理域（Customer-Supplier）**
- 经营决策域是Supplier，产出MRD
- 项目管理域是Customer，消费MRD创建项目
- 契约：`MRD.Approved`事件（RabbitMQ）

**2. 产研流程域 → 数字员工域（Customer-Supplier + ACL）**
- 产研流程域定义任务规范，数字员工域按规范执行
- 数字员工域对外部LLM API使用ACL（模型路由抽象层）

**3. 用户权限域 → 所有域（Published Language）**
- 统一发布用户/角色/权限的JSON Schema
- 所有域调用`iam-svc`验证身份与功能权限

**4. 基础设施域 → 数字员工域（Open Host Service）**
- 基础设施域提供Body运行时REST API
- 数字员工域调用API分配运行资源

**5. 自我进化域 → 所有域（Conformist）**
- 被动接收各域的改进需求事件
- 不主动调用其他域的服务

**6. 数字员工域 → 外部LLM API（Anti-Corruption Layer）**
- `agent-svc`内部`LLMRouter`组件隔离外部API差异
- 提供统一`generate()`接口，切换模型无需改业务代码

> **设计验证**：每个域的上下游关系已明确，跨域调用必须通过定义的映射模式，禁止直接数据库访问。
