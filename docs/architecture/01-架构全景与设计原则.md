## 1. 概述与范围

### 1.1 系统定位

产研数字军团OS 是面向 AI Native 时代的产研全流程研发体系。

它不是一套工具的堆叠，而是一个将 AI 能力嵌入组织底层的操作系统级平台。在这个体系下：

- AI 从"外挂"走向"内生"，成为业务设计、信息流转、流程编排和知识管理的基础环境
- 人与数字员工（AI Agent）协同工作，一个工程师驱动多个数字员工完成任务，人对最终交付结果承担责任
- 人想清楚、定义问题，数字员工探索解空间，人在价值岔路口做决策
- 经验资产化，隐性知识转化为显性资产，不随人员流动流失
- 全环节自我进化，从需求理解到代码生成，每个环节基于反馈持续优化

**产研数字军团OS 的本质是一个人机协同的分布式操作系统。**

它的三个关键特征：

**第一，以产研流程规范为核心抽象。** OS 定义"什么角色在什么条件下执行什么动作、产出什么、如何验收"——这个规范对人（工程师）和数字员工（AI Agent）一视同仁。产研流程覆盖从商机洞察到交付评估的全链路，任务流只是其中的执行层抽象。

**第二，意图驱动的执行模式。** 人负责想清楚和做决策，数字员工负责标准化执行、信息整合和流程推进。重复性工作由数字员工承担，人的价值向判断力、创造力和责任承担迁移。

**第三，经验资产化驱动价值。** 技术资产、知识条目、最佳实践在 OS 内持续沉淀与复用，形成组织复利。

### 1.2 业务目标

| 目标 | 核心指标 |
|------|---------|
| 支撑部门 AI Native 组织转型 | 四类角色与数字员工无缝配合；年内 50% 核心项目完成 AI Native 重构 |
| 应用软件部翻倍提效 | 需求交付周期缩短一倍；27 年 40% 人力支撑现有营业需求 |
| 协助部门经营模式转型 | 不增加人力完成自研产品交付；OS 作为对外业务的 AI 技术底座 |

### 1.3 四类角色

产研数字军团OS 服务四类角色。这四类角色是 AI Native 模式下公司内部已确定的职位体系，在 OS 中对应真实的权限边界和技能要求。

| 角色 | 定位 | 在 OS 中的核心活动 |
|------|------|-------------------|
| 经营管理者 | 全链路决策者 | 商机洞察、立项决策、经营分析、资源争取 |
| 复合型研发经理 | 经营与执行的枢纽 | 需求承接、任务分解、进度管控、交付收口 |
| AI 工程师 | 执行角色 | 利用 AI 工具与数字员工完成任务、跟进前沿技术 |
| AI Agent 架构师 | OS 建设者 | OS 能力迭代、数字员工优化、技术资产运营 |

**角色体系的可扩展性**

当前阶段优先服务内部四类职位。但 OS 的角色体系在设计上需支持后续扩展：
- 推广至其他公司时，可配置新的角色定义与权限边界
- 面向 C 端用户（如学校用户）时，可演进出完全独立的角色体系
- 角色、权限、技能要求均通过 iam-svc 动态配置，不硬编码

本期架构设计以内部四类角色为默认上下文，但所有涉及权限和角色的设计均需考虑扩展性。

## 2. 利益相关者与关注点

架构决策必须满足不同角色的关注点。本节显式列出每类利益相关者的核心关切，作为架构验证的基准。

| 利益相关者 | 角色定位 | 核心关注点 | 对应的架构质量属性 |
|------------|---------|------------|----------------------|
| **经营管理者** | 全链路决策者 | 经营数据准确完整；系统稳定不中断；决策效率；历史数据可追溯 | 数据一致性、可用性、安全性、可观测性 |
| **复合型研发经理** | 经营与执行枢纽 | 项目进度透明可见；任务分配有信息辅助，决策由人完成；交付质量可控；风险及早发现 | 可观测性、数据一致性、性能（看板加载）、可用性 |
| **AI 工程师** | 执行角色 | 任务规范清晰可执行；工具链流畅；Agent 执行结果可信；不被低效流程拖累 | 易用性、性能（任务下发延迟）、可靠性（结果不丢失） |
| **AI Agent 架构师** | OS 建设者 | 架构清晰可扩展；数字员工能力可配置；技术资产可沉淀复用；效果可度量 | 可扩展性、可维护性、可观测性、质量属性可验证 |
| **运维/部署负责人** | 系统运行保障（由 AI Agent 架构师承担） | 部署简单可重复；故障可快速定位；资源消耗可控；滚动升级不停机 | 可部署性、可观测性、可维护性、资源效率 |
| **安全合规负责人** | 安全与合规（由 AI Agent 架构师承担） | 数据不越权访问；操作可追溯；外部系统接入安全；LLM API Key 不泄露 | 认证与授权、审计日志、数据传输加密、密钥管理 |

**架构验证方法**：每个架构决策做出后，逐行核对本表——该决策是否损害了某一方的核心关注点？如果是，需要回退或补充缓解措施。

## 3. 架构驱动力

架构驱动力 = 业务目标 + 约束条件 + 质量属性需求。三者共同决定架构决策的边界。

### 3.1 业务目标

（详见 1.2，此处从架构视角重新表述）

| 业务目标 | 对架构的驱动作用 |
|---------|----------------|
| 支撑 AI Native 组织转型 | 架构必须原生支持"人+数字员工"双轨执行，不能把数字员工作为外挂功能处理 |
| 翻倍提效 | 架构必须支持高并发任务分配与执行状态实时同步，延迟直接影响提效比例 |
| 经营模式转型 | 当前阶段聚焦内部使用（不增加人力完成鸿合&智原等自研产品交付）；OS 需具备成为对外 AI 技术底座的能力基础——角色体系可配置、架构可扩展 |

### 3.2 约束条件

约束条件是不可协商的硬边界，架构必须在其范围内工作。

| 约束类型 | 具体约束 | 对架构的影响 |
|---------|---------|------------|
| **技术栈约束** | 团队主力技术栈为 Java，Spring Cloud 生态熟悉 | 服务端采用 Spring Cloud 全家桶（Gateway、Nacos、OpenFeign、RabbitMQ），降低学习成本 |
| **时间约束** | MVP 阶段 4 周完成 SC-01/02/10/12 四个场景 | 采用聚合服务策略（4+1 服务），避免 MVP 阶段拆分过细导致联调成本过高 |
| **预算约束** | LLM 调用费用需可控；不自建 GPU 集群 | 数字员工 MVP 阶段采用远程 LLM API 调用；引入 Token 消耗计量与配额管理 |
| **部署环境约束** | 内网环境，部分服务需支持自签名证书 | HTTP 客户端需支持跳过 SSL 验证（如 curl -k 场景）；Nacos/PostgreSQL 等中间件需支持内网部署 |
| **LLM 依赖约束** | 依赖外部 LLM API，可用性不受控 | 架构需支持模型切换（多 LLM Provider 路由）；关键节点必须有降级策略（人工接管） |
| **组织约束** | 当前团队规模小，运维能力有限 | MVP 阶段采用 Docker Compose 单机部署；避免引入 Kafka、Elasticsearch 等重依赖 |

### 3.3 质量属性需求

质量属性是架构的灵魂。本节用场景化方式描述，每个属性包含具体场景、刺激、预期响应。

#### 性能

| 场景 | 刺激 | 预期响应 | 优先级 |
|------|------|---------|--------|
| 个人工作台加载 | 用户打开个人工作台 | 首次加载 < 3s（MVP），< 1.5s（成熟期） | P0 |
| 任务分配推送 | Server 分配任务给数字员工 | 推送延迟 < 2s（MVP），< 500ms（成熟期） | P0 |
| 跨服务同步调用（P99） | pdm-svc 调用 iam-svc 查询用户信息 | < 500ms（MVP），< 200ms（成熟期） | P1 |
| 事件处理 | RabbitMQ 接收 `TaskInstance.Completed` 事件 | 处理速率 100/s（MVP），2000/s（成熟期） | P1 |

#### 可用性

| 场景 | 刺激 | 预期响应 | 优先级 |
|------|------|---------|--------|
| 单服务宕机 | pdm-svc 进程崩溃 | MVP：人工重启恢复；成熟期：自动重启 + 流量切换，恢复时间 < 30s | P0 |
| 数据库宕机 | PostgreSQL 主库崩溃 | MVP：人工介入恢复；成熟期：主从切换，切换时间 < 30s | P0 |
| 数字员工实例故障 | 数字员工执行中崩溃 | 标记失败，人工重试（MVP）；自动重试 3 次后转人工（成熟期） | P1 |
| LLM API 不可用 | OpenAI/Claude API 返回 503 | 自动切换备用模型提供商；无备用时任务标记为"等待 API 恢复" | P0 |

#### 安全性

| 场景 | 刺激 | 预期响应 | 优先级 |
|------|------|---------|--------|
| 未授权访问 | 无 Token 请求受限接口 | 拒绝访问，返回 401 | P0 |
| 越权访问 | 工程师 A 请求工程师 B 的任务详情 | 拒绝访问，返回 403；操作被记录到审计日志 | P0 |
| 敏感数据泄露 | 数据库文件被非法复制 | 敏感字段（API Key、Token）在数据库中加密存储（AES） | P0 |
| 数字员工越权 | 数字员工尝试访问超出任务范围的数据 | 数字员工 Token 绑定任务上下文，超出范围的操作被拒绝 | P1 |

#### 可扩展性

| 场景 | 刺激 | 预期响应 | 优先级 |
|------|------|---------|--------|
| 新增业务域 | 需要增加"财务域" | 新增独立服务，通过事件总线与现有域集成，无需修改现有服务代码 | P1 |
| 并发用户增长 | 在线用户从 50 增长到 500 | 核心服务支持水平扩展（K8s HPA），吞吐量线性增长 | P1 |
| 新 Agent 类型接入 | 需要接入新的数字员工类型（如测试专用 Agent） | 数字员工节点协议支持 Agent 类型注册与动态适配，无需修改 Server 端代码 | P2 |

#### 可观测性

| 场景 | 刺激 | 预期响应 | 优先级 |
|------|------|---------|--------|
| 数字员工行为审计 | 经营管理者询问"某数字员工上周做了什么" | 所有数字员工操作记录在审计日志中，可按数字员工实例 ID 检索，保留 ≥ 180 天 | P0 |
| 跨服务调用追踪 | 某个任务分配请求耗时异常 | 通过 Trace ID 串联所有跨服务调用，在 Grafana 中可视化调用链路 | P1 |
| 任务执行状态追踪 | 项目经理询问"某任务当前在哪个节点" | 任务状态变更事件完整记录，可通过任务 ID 查询完整状态流转历史 | P0 |

**每层定位一句话**：

- **Client 层**：面向人的交互界面，不直接访问微服务，统一经 API Gateway 接入
- **Server 层**：业务逻辑的权威执行地，按域拆分微服务，所有业务数据以Server为权威源
- **数字员工节点层**：Server 与 Agent 之间的通信桥梁，作为守护进程管理本地 Agent 的生命周期（启动/监控/停止），负责消息转发与协议适配，流式输出实时转发。任务调度由 Server 负责，任务执行由 Agent 完成，节点不承担任何业务职责

### 4.2 两条通信通道

#### 通道1：Client ↔ Server（同步请求/响应）

- **协议**：HTTPS + REST API（管理后台）；后期可引入 GraphQL（个人工作台聚合查询）
- **入口**：Spring Cloud Gateway（统一鉴权、路由、限流）
- **特点**：请求/响应模式，适用于用户主动操作（创建项目、查看状态、审批等）
- **鉴权**：JWT Token，Gateway 层统一验签，服务层不复验，只做数据范围过滤（org_id）

#### 通道2：数字员工节点 ↔ Server（WebSocket 长连接）

- **协议**：WebSocket
- **特点**：Server 主动推送任务给数字员工节点；数字员工节点主动上报执行结果和状态变更
- **选择 WebSocket 的理由**：
  - 走标准 HTTP 端口，部署简单，无需额外开端口和配代理
  - Spring Cloud Gateway 原生支持 WebSocket 路由
  - Rust（tokio-tungstenite）、Java（Spring WebSocket）、移动端均生态成熟
  - 通信场景为低频小消息的实时推送和上报，WebSocket 足够胜任
- **关键设计**：数字员工节点与 Server 之间保持长连接；节点断线后任务不丢失（Server 端维护待推送任务队列，节点重连后拉取）

### 4.3 通道 2 详细协议（数字员工节点 ↔ Server）

> **设计目标**：任务不丢失、可断线重连、可降级到人工接管
> **适用范围**：所有 Agent Node ↔ agent-svc 通信；任何违反本节的设计需在 ADR 中登记理由
> **状态**：v1.0-draft（2026-06-03）

#### 4.3.1 协议栈与消息格式

| 层     | 选型                                    | 理由                                |
| ----- | ------------------------------------- | --------------------------------- |
| 传输层   | WebSocket over TLS（默认 443 端口）         | 走标准 HTTP 端口，部署简单；与通道 1 共用 Gateway |
| 应用层协议 | JSON-RPC 2.0 风格（method + params + id） | 双向调用、心跳、推送、ACK 统一表达               |
| 序列化   | JSON，UTF-8，字段顺序不敏感                    | 工程实现成本最低                          |
| 单消息大小 | ≤ 1 MB                                | 超出走附件上传（对象存储）+ 引用传递               |

**消息分类**：

| 方向 | 消息前缀 | 示例 | 是否需要 ACK |
|---|---|---|---|
| Server → Node | `task.*`、`node.*`、`sync.*` | `task.assign`、`sync.snapshot` | 是 |
| Node → Server | `agent.*`、`node.*` | `agent.task_completed`、`node.heartbeat` | 否（at-least-once + 业务幂等） |
| 双向 | `node.*` | `node.heartbeat` / `node.heartbeat_ack` | 心跳不需 ACK，超时即视为断线 |

#### 4.3.2 握手与认证

**目标**：避免长期 Token 泄露；防止 Agent Node 仿冒

**握手流程**：

1. Agent Node 启动后，**先**用 `device_code`（设备唯一标识）+ `activation_code`（一次性激活码）调 REST `POST /api/agent-node/activate` 完成注册
2. Server 验证后，返回**短期 WS-Token**（默认 60 分钟）+ `refresh_token`（默认 7 天）+ `node_id`
3. Agent Node 携带 WS-Token 发起 WebSocket 升级：`Authorization: Bearer <ws_token>`
4. Gateway 透传到 agent-svc，后者用 Redis 验证 Token + node_id 匹配
5. 验证通过 → 建立长连接；失败 → 返回 401，Node 关闭连接

**Token 续期**：

- WS-Token 过期前 5 分钟，Server 通过当前连接推送 `node.token_refresh_required` 消息
- Node 用 refresh_token 调 `POST /api/agent-node/token/refresh` 换发新 WS-Token
- 换发成功后，旧 Token 立即失效；Node 用新 Token 重新建立连接（**不允许**在已建连接上"换证"）

**反仿冒**：

- `activation_code` 由用户在 OS 后台为特定 Node 生成，**一次性使用**
- 可选：Node ID + 客户端证书指纹双向校验（生产环境建议开启）

#### 4.3.3 心跳与断线判定

| 指标 | 值 | 说明 |
|---|---|---|
| ping 间隔 | 30 s | Node 主动发送 `node.heartbeat` |
| pong 超时 | 60 s | Server 60 s 未收到心跳即视为断线 |
| 断线后保留期 | 30 min | Node 上的任务实例进入 `AWAITING_NODE` 状态 |
| 保留期后动作 | 转人工 | 任务标记为 `NODE_OFFLINE_TIMEOUT`，触发人工接管流程 |

**断线后 Server 行为**（顺序执行）：

1. 立即停止向该 Node 推送新任务
2. 标记 Node 状态为 `OFFLINE`（持久化到 `agent_db`）
3. 该 Node 上 `RUNNING` 状态的任务实例 → `AWAITING_NODE`
4. 启动 30 min 计时器
5. 30 min 内重连 → 见 §4.3.5 重连补推
6. 超过 30 min → 任务实例标记 `NODE_OFFLINE_TIMEOUT`，发布领域事件 `Task.NodeOfflineTimeout`，订阅方（项目管理域）触发人工接管

#### 4.3.4 消息 ACK 与重试

**Server → Node 消息**：

- 每条业务消息带 `msg_id`（全局递增序列号，64 位）+ `require_ack=true`
- Node 收到后必须回 `ack {msg_id, status: "ok"|"rejected"}`
- Server 收到 ack 后从 `pending_messages[node_id]` 队列移除
- **未 ACK 重试**：Node 重连后，Server 重发所有未 ACK 的消息（去重靠 msg_id 客户端去重）

**Node → Server 消息**：

- **不带 ACK 要求**，走 at-least-once 投递（RabbitMQ 持久化）
- **幂等键**：每条 Node 上报的消息带 `client_msg_id`（Node 端生成的 UUID）
- **去重窗口**：Server 端用 Redis `SetNX` 去重，1 小时窗口
- **超过 1 小时重复消息**：直接丢弃，返回 `duplicate=true` 给 Node 端（仅作日志，不阻断）

**失败重试策略**：

| 失败类型 | Server 行为 | Node 行为 |
|---|---|---|
| 业务校验失败（如任务不存在） | 立即回 `ack {status: "rejected", reason}` | 记录日志，等待 Server 重派或人工 |
| Node 处理超时（> 任务 SLA × 2） | 标记任务 `TIMEOUT`，重派 | 主动停止执行 |
| Node 进程崩溃 | 见 §4.3.3 断线处理 | 重启后由 Server 补推 |

#### 4.3.5 任务重连补推

Node 重连后，Server 主动推送 `sync.snapshot` 消息，包含：

```json
{
  "msg_id": "...",
  "method": "sync.snapshot",
  "params": {
    "node_id": "...",
    "snapshot_at": "2026-06-03T10:00:00Z",
    "tasks": [
      {
        "task_id": "...",
        "instance_id": "...",
        "current_node": "wf_node_xxx",
        "status": "RUNNING",
        "context_ref": "minio://..."  // 完整上下文从对象存储拉取
      }
    ]
  }
}
```

**Node 端 reconciliation 逻辑**：

| 本地状态 | snapshot 中存在？ | Node 行为 |
|---|---|---|
| 有 | 有 | 继续执行（snapshot 是权威） |
| 有 | 无 | 任务已被 Server 收回，Node 主动停止并清理本地状态 |
| 无 | 有 | 新任务，Node 拉取完整上下文后开始执行 |
| 有 | 有但 current_node 不同 | 任务被其他 Node 接管（多 Node 协作场景），本 Node 停止 |

**MVP 简化**：MVP 阶段**禁止**多 Node 协作同一任务实例（一任务一 Node 绑定），简化上述逻辑。

#### 4.3.6 任务状态机（执行中 Node 崩溃场景）

任务在 Node 上的运行态：

```
         ┌──────────┐
         │  PENDING  │ （Server 已分配，Node 未开始）
         └────┬─────┘
              │ Node 开始执行（拉取上下文）
              ▼
         ┌──────────┐
         │ RUNNING   │ （Node 正在执行 LLM/工具调用）
         └────┬─────┘
              │ Node 上报完成 / 失败 / 阻塞
              ▼
   ┌──────────────────────┐
   │ REPORTED（终态前过渡）│ （Server 已收到上报，等待人工/自动验收）
   └──────────────────────┘
              │ 验收通过 / 失败
              ▼
   ┌──────────────────────┐
   │ COMPLETED / FAILED   │ （终态）
   └──────────────────────┘
```

**Node 崩溃场景恢复策略**：

| 崩溃发生点 | Node 重启后行为 | 已完成工作是否保留 |
|---|---|---|
| `PENDING` | 任务回到 `PENDING`，重新开始 | N/A |
| `RUNNING`（执行 LLM 调用） | 任务回到 `PENDING`，重新开始 | **否**（MVP 不做 checkpoint） |
| `RUNNING`（执行工具调用，如写文件） | 任务回到 `PENDING`，**但已写文件不回滚**（需业务侧幂等保证） | 部分 |
| `REPORTED` | Server 已收到，不重做 | 是 |

**MVP 简化**：MVP 阶段不实现 Node 端任务检查点（checkpoint），崩溃即重做已执行的 LLM/工具调用。MVP 后引入 LangGraph 风格的检查点机制，写入 `agent_checkpoint` 表。

#### 4.3.7 与 API Gateway 的关系

- **WebSocket 连接建立路径**：`Agent Node → wss://gateway/agent-ws → gateway-svc → inf-svc`
- **Gateway 责任**：
  - TLS 终结
  - 基础限流（单 IP 连接数 ≤ 10、单 Node QPS ≤ 100）
  - 透传消息（不解 JSON、不解析业务字段）
- **Gateway 不做的**：
  - 业务鉴权（由 inf-svc 验签 WS-Token）
  - 任务路由决策（由 agent-svc 决定）
  - 消息持久化（由 inf-svc 写 PostgreSQL/Redis）

**降级策略**：当 Gateway 检测到 inf-svc 不可用时，返回 `503 Service Unavailable` + `Retry-After`，Node 端按指数退避重连。

#### 4.3.8 协议版本化

- **握手时携带**：`protocol_version`（如 `"1.0.0"`）
- **Server 兼容性策略**：

| Node 版本 vs Server 支持范围 | 行为 |
|---|---|
| 在支持范围内（相同大版本） | 正常升级 |
| 低于最低支持版本 | 握手成功但立即推送 `node.upgrade_required`，Node 必须升级才能继续 |
| 高于最高支持版本 | 握手成功，Server 拒绝高版本特性，降级到共同支持版本 |

- **灰度发布**：Server 端可按 Node ID 维度开启/关闭新协议特性（如 `feature_flags.sync_snapshot_v2 = true`），无需重启

#### 4.3.9 监控指标与告警

**必采指标**：

| 指标 | 类型 | 说明 |
|---|---|---|
| `agent_ws_connections_active` | Gauge | 当前活跃 WS 连接数 |
| `agent_ws_messages_sent_total` | Counter | Server→Node 消息数，按 `method` 分桶 |
| `agent_ws_messages_received_total` | Counter | Node→Server 消息数，按 `method` 分桶 |
| `agent_ws_ack_latency_seconds` | Histogram | Server 推送后到收到 ACK 的延迟 |
| `agent_node_offline_duration_seconds` | Histogram | Node 断线时长分布 |
| `agent_node_reconnect_count_total` | Counter | Node 重连次数，按 `node_id` 分桶 |

**告警规则**（v1.0-draft，待 SRE 复核）：

| 规则 | 阈值 | 持续时间 | 级别 |
|---|---|---|---|
| 任意 Node `agent_node_reconnect_count_total` 1h 增长 > 5 | — | 立即 | P1（网络抖动） |
| 集群 `agent_ws_ack_latency_seconds` P99 > 5 s | — | 1 min | P1（消息积压） |
| `agent_ws_connections_active` 突降 > 50% | vs 1h 前 | 1 min | P0（系统性故障） |
| 任务 `NODE_OFFLINE_TIMEOUT` 1h 触发 > 3 次 | — | 立即 | P1（Node 稳定性） |

## 5. 架构设计原则

以下原则为指导所有架构决策的基础准则，任何架构决策不得违反。

### 原则一：可逆性优先

> 任何架构决策，优先选择容易回退的方案，而不是"理论上最优"的方案。

**理由**：AI Native 领域仍在快速演进，今天看来正确的决策（比如选某个 LLM 供应商、选某种 Agent 编排模式）可能在 6 个月后需要推翻。如果推翻的代价太高，架构就会成为枷锁。

**具体体现**：
- 模型路由抽象层：切换 LLM 供应商不需要改业务代码
- 数字员工节点协议版本化：升级节点时旧版本仍可工作，支持灰度发布
- 事件格式向后兼容：新增字段不影响旧消费者

### 原则二：渐进式演进，不一步到位

> 架构是长出来的，不是设计出来的。每个阶段做每个阶段合适的事。

**理由**：MVP 阶段团队小、需求不确定，过度设计（比如直接拆 12 个微服务）只会拖慢交付速度。等域边界在实际开发中被验证后，再拆分。

**具体体现**：
- MVP：4+1 聚合服务（pdm-svc 聚合 4 个高频联动域）
- 成长期：按域拆分为 8 个服务
- 成熟期：12 个独立服务，每个域独立演进
- 每个阶段的拆分触发条件在架构文档中显式定义（见 02-服务端架构设计 2.1 节）

### 原则三：域边界清晰，跨域调用必须经过显式契约

> 域与域之间不允许直接访问对方数据库；跨域通信必须通过 API 契约或领域事件。

**理由**：如果允许跨域直接查库，域边界就会逐渐腐蚀，最终回到"大泥球"架构。

**具体体现**：
- 同步调用：通过 OpenFeign，调用方需处理超时和熔断
- 异步事件：通过 RabbitMQ，事件格式在《04-架构决策记录》中定义
- 数据库：core_db 内多 Schema 隔离，禁止跨 Schema 直接访问；独立库（iam_db、asset_db、agent_db）完全隔离

### 原则四：OS 不自动决策，关键节点必须人工确认

> AI 可以推荐、可以执行、可以建议，但最终决策权必须在人手里。

**理由**：这是 PRD 中反复强调的设计约束。如果 OS 自动做经营决策、自动立项、自动交付，一旦 AI 出错，损失不可挽回。

**具体体现**：
- 经营决策域：MRD 审批、立项决策必须由经营管理者确认
- 项目实施域：任务完成验收必须由人审核（审核节点）
- 数字员工域：数字员工只能执行被分配的任务，不能主动发起任务

### 原则五：对人和数字员工一视同仁

> 任务和产研流程规范对人（工程师）和数字员工（AI Agent）使用同一套抽象，不两套系统。

**理由**：如果人和数字员工走不同的流程，就会出现"数字员工走捷径、人走繁文缛节"或者反过来，最终导致流程被绕过。

**具体体现**：
- 任务规范（Task Spec）同时适用于人和数字员工
- 执行工作台：人使用的界面和数字员工使用的 API 基于同一套任务规范
- 验收标准：人和数字员工的产出使用同一套验收标准

### 原则六：声明 vs 证据

> 区分"口头承诺"（声明）和"可验证成果"（证据），OS 只认证据。

**理由**：PRD 中强调的方法论。如果只记录"我说我做完了"，进度监控就毫无意义。必须由可验证的交付物（代码提交、测试报告、设计文档）来证明任务完成。

**具体体现**：
- 任务提交时必须关联证据（Git Commit ID、文档 URL、测试报告）
- 数字员工执行结果必须包含执行日志和产出物引用
- 进度报表区分"声明进度"和"证据进度"

### 原则七：经验资产化的前提是结构化

> 技术资产和知识经验只有在结构化之后才能被复用。散落在聊天记录和个人脑中的经验不是资产。

**理由**：很多团队说"我们重视知识管理"，但最后知识库变成文件堆砌，没人能找到有用的信息。

**具体体现**：
- 技术资产必须有元数据（适用场景、技术栈、依赖条件）
- 知识经验必须关联到具体的任务规范（知道"在什么场景下有用"）
- 复用行为可被追踪（被引用次数、被采纳次数，驱动价值积分）
