## 1. 概述

### 1.1 Client 层定位

产研数字军团OS 的 Client 层是面向人的交互界面，是所有四类角色与系统交互的唯一入口。

Client 层不直接访问微服务，统一经 API Gateway 接入 Server 层。Client 不承载任何业务逻辑，仅负责展示、交互和数据校验。

### 1.2 客户端产品矩阵

OS 的 Client 层按使用者分为两个独立产品，后期规划移动端：

| 产品 | 使用者 | 核心定位 | 端形态 |
|------|--------|---------|--------|
| **应用端** | 经营管理者、复合型研发经理、AI 工程师 | 产研全流程执行与协作 | PC / Web（MVP），后期桌面端（Electron/Tauri） |
| **管理后台** | AI Agent 架构师 | OS 能力建设与系统运维 | PC / Web |
| **移动端**（规划中） | 全角色 | 审批 + 消息通知 | Mobile App（Android / iOS） |

**为什么分两个前端项目而不是一个多角色项目**：

- 目标用户不同：应用端面向产研业务人员，管理后台面向系统建设者
- 功能范围几乎不重叠：应用端是项目管理/任务执行/进度看板，管理后台是原型管理/权限配置/节点监控
- 权限模型差异大：应用端按项目制做数据隔离，管理后台按系统功能做功能隔离
- 独立部署节奏不同：应用端迭代频率高（跟随业务需求），管理后台迭代频率低（跟随架构演进）

### 1.3 技术选型约束

MVP 阶段的选型原则：

- 团队前端熟悉度优先，不追新技术
- PC / Web 优先，桌面端和移动端后做
- 两个前端项目尽量统一技术栈，降低维护成本

## 2. 应用端架构设计

### 2.1 功能模块总览

应用端按角色提供不同功能视图，但底层是同一个产品。

#### 经营管理者视图

经营数据看板 — 营收、资源利用率、项目健康度等关键指标的可视化

商机管理 — 商机录入、商机评估、商机转化为项目立项

立项审批 — MRD 审阅、立项决策审批（人工确认节点）

经营分析 — 项目投入产出分析、资源负载分析、趋势报表

资源管理 — 部门资源池查看、跨项目资源分配概览

#### 复合型研发经理视图

项目管理 — 项目列表/看板、项目详情（里程碑/风险/成员）、项目状态汇报

需求管理 — 需求承接（从商机分解）、需求理解确认、需求变更管理

任务管理 — 任务看板（按状态/负责人/优先级）、任务分配（有信息辅助，人做决策）、任务进度监控（声明 vs 证据）、任务验收审核

进度管控 — 项目进度视图、风险预警列表、需求变更追踪

交付收口 — 交付物检查清单、交付验收确认、效果评估

#### AI 工程师视图

个人工作台 — 待办任务列表、进行中任务、已完成任务

任务执行 — 任务详情（规范/上下文/验收标准）、执行过程记录、产出物提交（关联证据）

数字员工协同 — 数字员工执行状态查看、执行结果确认/打回、数字员工输出流式预览

技术资产 — 技术资产库浏览、资产检索与引用、资产贡献

知识体系 — 知识条目查阅、个人知识收藏

#### 共同模块

消息中心 — 系统通知、预警提醒、@消息、企业微信推送配置

组织/项目切换 — 一个人属于一个部门但参与多个项目，支持快速切换项目上下文

个人设置 — 账号信息、通知偏好、语言/主题

### 2.3 前后端交互规范

- **接口风格**：REST API（MVP），后期引入 GraphQL（聚合查询场景）
- **鉴权**：JWT Token，登录后存储在 localStorage，每次请求携带 Authorization Header
- **数据权限**：由 Server 端各业务域自行管理（按 org_id / project_id 过滤），Client 端不缓存完整数据集
- **实时推送**：通过 SSE（Server-Sent Events）接收通知和状态变更推送，WebSocket 通道留给数字员工节点使用
- **分页与加载**：列表接口统一分页参数（page / page_size / sort），大列表支持虚拟滚动
- **错误处理**：统一错误码映射，网关层 401 自动跳转登录，业务错误由页面内提示

### 2.4 安全要求

- Token 不暴露在 URL 参数中
- 敏感操作（删除、审批）需二次确认
- XSS 防护：所有用户输入渲染前转义
- CSRF 防护：POST/PUT/DELETE 请求携带 CSRF Token（如果使用 Cookie 鉴权方案）

## 3. 管理后台架构设计

### 3.1 功能模块总览

管理后台面向 AI Agent 架构师，负责 OS 能力的建设、配置与运维。

#### 数字员工管理

原型管理 — 原型列表/详情/创建/编辑/发布，原型包含角色定义、职责范围、技能清单、工具使用清单、初始记忆、运行环境定义、协议适配描述

原型版本管理 — 原型迭代历史、版本对比、回滚

技能库 — 技能定义与管理（技能与原型的关联关系）

#### 组织与权限管理

组织架构 — 部门树管理（增删改查）、人员归属维护

项目矩阵 — 项目创建/维护、项目成员管理（一个人属于一个部门但参与多个项目）

角色配置 — 角色定义（角色名、权限范围、技能要求），支持动态配置不硬编码

功能权限 — 菜单/按钮级权限分配（RBAC），按角色授权

数字员工实例 — 实例列表（属于哪个项目、部署在哪个节点、运行状态）、实例创建/停用

#### 系统配置

LLM Provider 路由 — 模型提供商配置（API Key、Endpoint、模型列表）、路由规则（默认模型/降级模型）

Token 配额管理 — 按用户/按项目的 LLM Token 消耗限额

节点管理 — 数字员工节点列表（注册状态、在线状态、心跳信息）、激活码生成

事件订阅规则 — 哪些事件推送给哪些人、推送渠道（站内/企业微信）

系统参数 — 全局配置项管理

#### 运维监控

节点状态监控 — 数字员工节点在线/离线状态、心跳延迟、连接数

任务执行监控 — 任务队列深度、推送延迟、执行成功率

系统日志 — 审计日志查询、操作日志查询

### 3.3 与应用端的差异

| 维度 | 应用端 | 管理后台 |
|------|--------|---------|
| 使用频率 | 高（日常使用） | 低（配置/运维时使用） |
| 交互复杂度 | 高（看板拖拽、流式输出、复杂表单） | 中（表格 CRUD、配置表单） |
| 数据量 | 按项目过滤，单项目数据量可控 | 全局数据，需分页/搜索 |
| 权限模型 | 项目制数据隔离 + 功能权限 | 系统级功能隔离（管理权限） |
| 实时性要求 | 高（任务状态变更需即时刷新） | 低（配置变更不需要实时推送） |

## 4. 移动端规划

### 4.1 定位

移动端是 PC/Web 端的功能子集，聚焦两个场景：

审批流程 — 立项审批、验收确认、需求变更审批等需要人工确认的关键节点

消息通知 — 系统预警、任务完成通知、@消息推送

### 4.2 不做的

不在移动端做完整的任务管理、项目管理、技术资产管理等复杂操作——这些需要大屏幕和键盘输入，移动端体验不佳且容易出错。

## 5. 部署方案

### 5.1 MVP 阶段

应用端和管理后台均为纯静态资源（SPA），部署方式：

Nginx 托管静态文件 + 反向代理到 API Gateway

或直接由 Spring Cloud Gateway 提供静态资源服务（MVP 简化方案）

### 5.2 成长期

前后端分离部署，CI/CD 流水线：

前端构建产物上传至 CDN / 对象存储

Nginx 托管 + 反向代理到 API Gateway

支持灰度发布（通过 Nginx分流或 Gateway 路由规则）
