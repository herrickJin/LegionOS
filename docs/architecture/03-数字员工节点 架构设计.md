### 1.1 数字员工节点 定位与职责边界

数字员工节点 本质是运行在终端上的 代理通信中间件 ，凌驾于所有本地 Agent 之上，是 Server 与 Agent 之间的唯一通信桥梁。

**一句话定位**：是数字员工的本地监护人——它替 Server 管理本地 Agent 进程的生死，替 Agent 翻译 Server 的指令，替 Server 转发 Agent 的产出。数字员工节点 + 数字员工Agent = body

**做什么**：

1. **实例初始化**：解析 Server 下发的数字员工原型配置，选择对应 Driver，启动 Agent 子进程
2. **进程管理**：管理 Agent 子进程的全生命周期——启动、监控、优雅停止、异常回收
3. **协议翻译**：把 Server 的 JSON-RPC 消息翻译成 Agent 能理解的调用方式，把 Agent 的输出翻译回 Server 的消息格式
4. **消息路由**：双向消息分发——Server 任务推送到对应 Agent 实例，Agent 产出上报到 Server
5. **流式转发**：Agent 的流式输出（SSE/chunked/逐行 stdout）实时转发给 Server，不做攒批
6. **连接保活**：与 Server 保持 WebSocket 长连接，心跳、断线重连、Token 续期

### 1.2 设计原则

**原则一：守护进程身份**

数字员工节点 是系统级守护进程，不是用户态工具。开机自启、崩溃自重启、对用户透明。工程师不需要手动启动 ACP，也不需要知道 ACP 在做什么——他们只跟 Agent 交互。

**原则二：协议翻译，不做业务**

数字员工节点 的每一行代码都应该在"翻译"或"路由"，如果出现业务判断逻辑（如"这个任务结果是否合格"），说明职责越界了。

**原则三：Agent 无感**

Agent 不需要知道自己通过 数字员工节点 与 Server 通信。对 Agent 来说，它要么收到 CLI 参数/stdin 输入，要么收到 HTTP 请求——跟直接被人操作没有区别。协议差异由 数字员工节点 的 Driver 抹平。

**原则四：不落盘，崩溃即重做**

本地零持久化。数字员工节点 崩溃后重启，从 Server 拿 snapshot 恢复状态。好处：无状态意味着无脏状态，无需实现 WAL/checkpoint，简化实现和排障。

**原则五：Rust 选型理由**

| 考量    | 选择 Rust 的理由                                                       |
| ----- | ----------------------------------------------------------------- |
| 内存安全  | 守护进程需 7×24 运行，无 GC 意味着无 GC pause，无数据竞争意味着无野指针                     |
| 跨平台   | 同一份代码交叉编译到 macOS / Linux / Windows，后期可编译到 Android / iOS           |
| 零成本抽象 | Driver trait 的动态分发在热路径上零开销，tokio 异步运行时轻量高效                        |
| 生态成熟  | tokio-tungstenite（WS）、serde_json（序列化）、tracing（日志）、clap（CLI）均为一等公民 |
| 单二进制  | 编译产物是单个静态链接二进制，无运行时依赖，安装即用                                        |

### 1.3 技术选型与依赖

Rust 语言本身的选型理由见 §1.2 原则五。本节聚焦具体 crate 选择、各模块的依赖关系，以及明确不选的方案和理由。
#### 1.3.1 工具链

- Rust Edition 2021，stable 频道，不依赖 nightly 特性——ACP 是基础架构组件，稳定性优先
- MSRV（Minimum Supported Rust Version）为当前最新 stable 版本
- `cargo` 构建系统
- `clippy` 静态检查，CI 强制通过
- `rustfmt` 代码格式化，统一风格
- 交叉编译：`cargo-zigbuild`（macOS ARM → Linux x86_64 单命令），`cross`（Docker 化备选）

#### 1.3.2 异步运行时：tokio

选择 tokio（full features），不选 async-std / smol。

理由：
- 生态最成熟，tokio-tungstenite、reqwest、hyper 均基于 tokio 构建
- `tokio::process` 直接支持异步子进程 spawn、stdin/stdout 管道管理，无需额外依赖
- `tokio::signal` 支持优雅关闭（SIGTERM/SIGINT）
- 工作窃取调度器适合 ACP 的并发模型：1 个 WebSocket 连接 + N 个 Agent 子进程管道 + M 个 HTTP 请求

tokio feature flags：
- `full`：含 rt-multi-thread、process、signal、io-util、sync、time 等
- `tracing`：与 tracing crate 集成，自动为异步任务注入 span

#### 1.3.3 核心依赖清单

按关注点分组：

WebSocket 连接
- tokio-tungstenite：WebSocket 客户端，基于 tokio 异步 IO
- rustls：TLS 加密传输，纯 Rust 实现，零系统依赖（不选 native-tls）

序列化与协议
- serde + serde_json：JSON-RPC 2.0 消息的序列化/反序列化
- serde_derive：derive 宏（`#[derive(Serialize, Deserialize)]`）

HTTP 客户端
- reqwest：Token 续期（REST API）、v2 vault 密钥拉取、HttpApiDriver 上游请求。基于 tokio + rustls，与 WS 栈共用 TLS 配置

进程管理
- tokio::process（标准库模块）：异步子进程 spawn、stdin/stdout 管道读写
- tokio::signal：Unix 信号捕获（SIGTERM/SIGINT），Windows 用 `SetConsoleCtrlHandler`

日志与遥测
- tracing：结构化日志 + span 追踪。比 `log` crate 更适合守护进程的多任务并发场景——span 可追踪一个任务从 WebSocket 入站到 Agent 出站的完整生命周期（跨越多个 await 点）
- tracing-subscriber：日志格式化 + 输出（stdout + 文件）
- tracing-appender：日志文件轮转（按大小和文件数上限滚动）

错误处理
- thiserror：库级错误类型（`derive(Error)`），类型精确，模式匹配友好
- anyhow：应用级错误处理（main 函数、Driver 实现），减少样板代码

CLI 与配置
- clap：命令行参数解析（derive 模式，`#[derive(Parser)]`，类型安全）
- toml：`acp.toml` 配置文件解析
- dirs：跨平台用户目录获取（`~/.legionos/`、`%APPDATA%\LegionOS\`）

异步 trait
- async-trait：为 `AgentDriver` trait 的异步方法提供编译期支持（`#[async_trait]` 宏），直到 Rust 原生 async fn in trait 稳定

#### 1.3.4 模块与依赖对应关系

以下列出每个模块直接依赖的外部 crate，不含 Rust 标准库：

- tokio-tungstenite、rustls：WebSocket 连接管理

ProcessManager
- tokio::process：异步子进程 spawn、stdin/stdout 管道读写、SIGTERM/SIGKILL

#### 1.3.5 明确不选的方案及理由

rustls，不选 native-tls
- native-tls 依赖系统 OpenSSL / LibreSSL / Security.framework，不同平台行为不一致，版本碎片化
- rustls 纯 Rust 实现，交叉编译不需要目标平台安装 SSL 库，单二进制无运行时依赖
- 性能相当，rustls 已被 Google、AWS 等广泛采用

tokio，不选 async-std
- async-std 生态厚度不及 tokio——tokio-tungstenite、reqwest、hyper 均基于 tokio
- `tokio::process` 对子进程管理的支持更成熟（async-std 的子进程 API 较简陋）
- tokio 有 `tokio-console` 用于生产环境异步任务诊断，对守护进程排障极有价值

TOML，不选 YAML / JSON 作配置文件格式
- TOML 是 Rust 生态配置文件的事实标准（cargo 自身用 TOML）
- YAML 的缩进敏感和隐性类型转换（`yes`→bool、`1.0`→float）在生产配置中容易引入意外错误
- JSON 不支持注释，不方便人工手写和维护

tracing，不选 log 作日志方案
- `log` 只有日志级别（error/warn/info/debug/trace），没有 span 和结构化字段
- tracing 的 span 能追踪一个 WebSocket 请求从入站到出站的完整生命周期，跨越多个 await 点不丢失上下文
- tracing 与 tokio 深度集成，`#[tracing::instrument]` 自动为异步函数注入 span
- 守护进程的日志量巨大，没有 span 的关联能力，排障基本靠猜

thiserror + anyhow，不选单一错误处理方案
- thiserror 用于库级错误（模块间传递），类型精确，支持 `match` 模式匹配
- anyhow 用于应用级错误（main 函数、Driver 实现），减少 `Box<dyn Error>` 样板
- 两层分工是 Rust 社区共识实践（参考 std、tokio、reqwest 等主流库的做法）

### 2.2 模块职责说明

#### 1. ConnectionMgr（连接管理器）

**一句话职责**：管理 ACP 与 Server 之间的 WebSocket 连接全生命周期。

**上下文**：ACP 是守护进程，7×24 在线。但它和 Server 之间隔着一层网络——网络会断，Token 会过期，Server 会主动踢线。ConnectionMgr 就是把"和 Server 保持连线"这件事封装成可靠能力的模块。

具体做什么：

- **连接建立**：启动时携带 WS-Token 发起 WebSocket 升级，TLS 由 rustls 终结
- **心跳保活**：每 30s 发 `node.heartbeat`，60s 无响应视为断线
- **断线重连**：指数退避（1s → 2s → 4s → 8s → 最大 60s），重连后触发 sync.snapshot 对账
- **Token 续期**：收到 `node.token_refresh_required` 后调 REST 换新 Token，重建连接
- **消息收发**：收 Server 消息交给 MessageRouter，发 Agent 上报消息到 Server（含 `client_msg_id` 幂等键）

**关键设计决策**：所有连接中断都由它统一处理，上层模块（ProcessManager、MessageRouter）不需要关心"现在有没有连上"——reconciliation 交由 sync.snapshot 完成，断开期间的输出缓存在 channel buffer，溢出后丢弃进度消息、保留终态。

#### 2. MessageRouter（消息路由器）

**一句话职责**：双向消息分发——Server 消息路由到对应模块，Agent 产出路由到 Server。

**上下文**：ACP 同时在处理多种消息——instance.deploy、task.assign、node.heartbeat_ack、sync.snapshot。这些消息的目标不同：有的是给 Driver 的，有的是给 ConnectionMgr 的，有的是给 ProcessManager 的。MessageRouter 存在的意义是：**只做"找到目标"，不做"理解内容"**。它不解析任务参数、不判断任务是否合理——那是 Driver 和 Server 的事。

路由逻辑：

- `instance.*` → 实例生命周期操作，路由到 ProcessManager（deploy/terminate）
- `task.*` → 按 `instance_id` 查找 Agent 实例，路由到对应的 Driver
- `node.*` → ACP 自身消息（心跳、Token 续期、配置更新），内部处理
- `sync.*` → 触发 reconciliation 对账

**关键设计决策**：路由表是内存中的 `HashMap<instance_id, AgentHandle>`，崩溃即清空，重连后从 Server 的 sync.snapshot 重建。零持久化，无脏路由。


#### 3. DriverRegistry（Driver 注册表）

**一句话职责**：管理已注册的 Agent Driver，提供统一的 Driver 查找和调用接口。

**上下文**：ACP 需要支持多种 Agent——自研的 Hermes、Anthropic 的 Claude Code、通用的 HTTP API Agent。它们的启动方式、通信协议、输出格式各不相同。DriverRegistry 把所有差异封装在 `AgentDriver` trait 后面，上层模块（ProcessManager、ProtocolAdapter）只跟 trait 交互，不关心具体实现。

MVP 阶段硬编码 3 种 Driver：

- **HermesDriver**：CLI 子进程（stdin/stdout 管道），stdout 输出 JSON 流
- **ClaudeCodeDriver**：CLI 子进程（stdin/stdout 管道），stdout 输出 stream-json
- **HttpApiDriver**：HTTP POST + SSE 流式响应

Driver trait 统一接口：

- `driver_type()` — 返回类型标识，对应 `instance.deploy` 中的 `driver_type`
- `deploy(config)` — 部署实例（spawn 子进程或 HTTP 初始化）
- `terminate(handle)` — 终止实例
- `send_task(handle, task)` — 下发任务
- `subscribe_output(handle)` — 订阅流式输出

**关键设计决策**：硬编码而非插件化。MVP 不需要动态加载——agent-svc 知道能用哪几种 Agent，ACP 直接编译进去即可。后期需要扩展时再引入 trait object 动态加载机制。

#### 4. ProtocolAdapter（协议适配层）

**一句话职责**：Server 的 JSON-RPC ↔ Agent 原生协议的双向翻译。

**上下文**：Server 说的是"JSON-RPC 2.0 标准消息"，Agent 说的是各自的"方言"——Hermes 说 CLI 参数 + stdin JSON，Claude Code 说不同的 CLI 参数 + stdin 文本，HTTP Agent 说 REST + SSE。ProtocolAdapter 就是翻译层：Server 下发的 `instance.deploy`（含 `InstanceDeployConfig`）翻译成各 Driver 能消费的格式；Agent 的流式输出翻译回 Server 的 `agent.task_progress` / `agent.task_completed` 消息。

具体翻译：

- **Server → Agent**：`instance.deploy` 的 `agent_config.system_prompt/model/tools` 映射为 CLI 参数或 HTTP JSON body
- **Agent → Server**：Agent stdout 逐行 / SSE chunk → `agent.task_progress`（逐 chunk 转发，不攒批）；最终输出 → `agent.task_completed`

**关键设计决策**：流式逐 chunk 转发，不做攒批。Agent 吐一行，ACP 立即转一行。`task_progress` 消息不要求 ACK，at-least-once + 业务幂等。背压策略：channel buffer 溢出时丢弃最旧的 progress，保留终态。#### ProcessManager（进程管理器）

管理 Agent 实例的全生命周期——从部署到终止

| 职责 | 说明 |
|------|------|
| 实例部署 | 收到 `instance.deploy` 后，根据 `InstanceDeployConfig` 选择 Driver、spawn 子进程（CLI 类型）或建立 HTTP 长连接（HTTP 类型），注入 credentials 到环境变量 |
| 实例终止 | 收到 `instance.terminate` 后，优雅停止 Agent（SIGTERM → 等待 10s → SIGKILL），清理资源，上报 `agent.instance_stopped` |
| 健康监控 | 监控子进程退出码；HTTP Agent 通过定期 probe 检测存活性 |
| 异常回收 | Agent 崩溃后，清理资源、从路由表移除，向 Server 上报 `agent.task_failed` + `agent.instance_stopped`，由 Server 决定是否重新 deploy |
| 资源清理 | Agent 退出后清理子进程资源（关闭 stdin/stdout 管道、回收 PID、清理临时工作目录） |

**实例生命周期**：

```
instance.deploy → DEPLOYING → RUNNING ──→ task.assign（可多次）──→ instance.terminate → TERMINATING → STOPPED
                                   ↑              │
                              部署成功         Agent 执行任务
                              上报 started     完成后等下一个

异常路径：
RUNNING → Agent 崩溃 → CRASHED → 上报 instance_stopped → Server 决定是否重新 deploy
```

**进程管理策略**：

```
Server 下发 instance.deploy
  ↓
ProcessManager 根据 driver_type 从 DriverRegistry 获取 Driver
  ↓
Driver.deploy(InstanceDeployConfig) → spawn Agent 子进程
  ↓
部署成功 → 上报 agent.instance_started → 实例进入 RUNNING 状态
  ↓
Server 下发 task.assign（可多次）
  ↓ ProcessManager 检查该 instance_id 是否已有运行中的 Agent
  ├─ RUNNING → 复用现有进程，通过 Driver.send_task() 发送任务
  ├─ CRASHED → 返回错误，提示 Server 重新 deploy
  └─ STOPPED（不存在）→ 返回错误，提示 Server 先 deploy

Server 下发 instance.terminate
  ↓
ProcessManager 调用 Driver.terminate()
  ├─ CLI Agent → SIGTERM → 等待 10s → SIGKILL
  └─ HTTP Agent → 关闭连接
  ↓
清理资源 → 上报 agent.instance_stopped { reason: "terminated" }

Agent 子进程意外退出
  ↓
ProcessManager 检测退出码
  ├─ exit 0 → 可能是任务完成后 Agent 自行退出，上报 instance_stopped { reason: "exited_normally" }
  ├─ exit 非0 → 异常退出，上报 task_failed + instance_stopped { reason: "process_exit", exit_code }
  └─ 被 SIGKILL → 上报 task_failed + instance_stopped { reason: "process_killed" }
```

**安全边界**：
- Agent 子进程以当前用户权限运行，ACP 不做权限提升
- Agent 子进程继承 ACP 的环境变量白名单（PATH、HOME 等），不继承敏感变量
- credentials（API Key 等）通过 `InstanceDeployConfig.credentials` 注入，仅注入到 Agent 子进程的环境变量中，ACP 自身不存储
- Agent 子进程的工作目录由 `InstanceDeployConfig.working_dir` 决定，默认为 `$HOME/.legionos/work/{instance_id}/`


#### 5. ProcessManager（进程管理器）

**一句话职责**：Agent 实例的全生命周期管理——部署、监控、优雅停止、异常回收。

**上下文**：这是 ACP 作为"守护进程"最核心的能力。ACP 之所以叫"数字员工监护人"，就是因为 ProcessManager 替 Server 管理着本地 Agent 进程的生死。Server 说"启动一个代码审查员"，ProcessManager 根据 `InstancedeployConfig` 选 Driver、spawn 进程、注入凭证；Server 说"停掉这个实例"，ProcessManager 优雅终止；Agent 自己崩溃了，ProcessManager 检测退出码、清理资源、上报 Server。

核心操作：

- **实例部署**：收到 `instance.deploy` → 根据 `driver_type` 获取 Driver → `Driver.deploy(InstanceDeployConfig)` → spawn 子进程 / HTTP 初始化 → 上报 `agent.instance_started`
- **实例终止**：收到 `instance.terminate` → SIGTERM → 等 10s → SIGKILL → 清理资源 → 上报 `agent.instance_stopped`
- **健康监控**：监控子进程退出码；HTTP Agent 定期 probe `/health`
- **异常回收**：Agent 崩溃 → 清理资源 → 从路由表移除 → 上报 `task_failed` + `instance_stopped` → Server 决定是否重新 deploy

**关键设计决策**：实例生命周期（deploy/terminate）和任务执行（assign/cancel）是两层独立操作。先 deploy 实例，再向实例 assign 任务，一个实例可接收多个任务。这样避免了"每个任务都要冷启 Agent"的性能浪费。
#### 6. ConfigManager（配置管理器）

**一句话职责**：ACP 自身的配置管理——加载、解析、运行时更新。

**上下文**：ACP 作为守护进程，需要知道 Server 地址、日志级别、心跳间隔、Agent 可执行文件路径、设备标识等配置。这些配置来源多样——本地 TOML 文件、CLI 参数、环境变量、Server 运行时覆盖。ConfigManager 统一管理这些配置的加载优先级和运行时更新。

配置项：

- Server 地址：`wss://legionos.example.com/agent-ws`
- 日志级别：`info`（可被 CLI `--log-level` 覆盖）
- 心跳间隔：30s（可被 Server `node.config_update` 覆盖）
- 重连策略：指数退避最大 60s，无限重试
- Agent 可执行文件路径：`/usr/local/bin/hermes`、`claude`（$PATH）
- 设备标识：首次启动生成 UUID v4

**关键设计决策**：配置优先级链——CLI > Server 运行时覆盖 > 环境变量 > 配置文件 > 编译期默认值。Server 覆盖仅存内存，不写回文件，确保 ACP 重启后行为可预测，避免"配置漂移"。

---

#### 7. SecretManager（密钥管理器）

**一句话职责**：管理 Agent 运行所需的密钥和凭证——提取、注入、释放。

**上下文**：Agent 运行时需要 API Key（如 ANTHROPIC_API_KEY）才能调用模型。这些凭证从哪来、怎么给 Agent、什么时候释放——SecretManager 就是管这件事的。

MVP 阶段极简：Server 在 `instance.deploy` 的 `credentials` 字段直接下发明文凭证，ACP 提取后注入到 Agent 子进程的环境变量中，Agent 启动后 ACP 立即释放内存中的明文引用。

MVP 流程：

Server 下发 `credentials: { ANTHROPIC_API_KEY: "sk-ant-..." }` → ACP 提取 → 注入 Agent 环境变量 → 释放引用 → Agent 直接使用

v2 演进为 Vault 引用模式：Server 不再接触明文，只下发 `vault://secrets/anthropic-key`，ACP 从密钥管理服务拉取明文注入。

**关键设计决策**：凭证明文仅在 `resolve_credentials` 调用期间存在于 ACP 内存，注入后即释放。MVP 凭证明文经 TLS 加密的 WebSocket 传输，由 Server 端控制有效期。ACP 自身不落盘不存储。

### 2.3 模块依赖关系

```
ConfigManager ──→ ConnectionMgr（Server 地址、心跳间隔、重连策略）
               ──→ ProcessManager（Agent 可执行文件路径）
               ──→ DriverRegistry（Driver 注册配置）
               ──→ SecretManager（v2：vault 服务地址）

ConnectionMgr ──→ MessageRouter（收到的消息交给路由分发）
               ←── MessageRouter（发出的消息通过连接发送）

DriverRegistry ──→ ProcessManager（根据 agent_type 查找 Driver）
                ──→ ProtocolAdapter（根据 agent_type 选择翻译策略）

ProcessManager ──→ MessageRouter（Agent 退出通知 → 上报 Server）
                ──→ ProtocolAdapter（Agent 子进程的 stdin/stdout 管道）
                ──→ SecretManager（deploy 时调用凭证解析，注入环境变量）

ProtocolAdapter ←── MessageRouter（Server 消息需要翻译后才能发给 Agent）
                ──→ MessageRouter（Agent 输出翻译后交给路由发往 Server）
```

## 3. 通信设计

### 3.1  节点 ↔ Server 通信

本节从 节点 Rust 端视角描述，协议细节参见 01 §4.3。
#### 3.1.1 连接生命周期

```
ACP 启动
  │
  ├─ 1. 读取本地配置（server_url, device_code）
  ├─ 2. 检查是否有有效 WS-Token（本地不持久化，首次启动无 Token）
  ├─ 3. 无 Token → 调 REST POST /api/agent-node/activate（device_code + activation_code）
  │      → 获取 ws_token + refresh_token + node_id
  ├─ 4. 携带 ws_token 发起 WebSocket 升级
  ├─ 5. 连接成功 → 启动心跳定时器 + 消息收发循环
  │
  │  ...正常运行...
  │
  ├─ 6. 收到 node.token_refresh_required → 调 REST 换发新 Token → 重建连接
  ├─ 7. 连接断开 → 指数退避重连 → 重连成功后收到 sync.snapshot → reconciliation
  └─ 8. 收到 shutdown 信号 → 优雅关闭所有 Agent → 断开连接 → 退出
```

#### 3.1.2 数字员工节点 端消息处理

| 收到 Server 消息 | ACP 行为 |
|-----------------|---------|
| `instance.deploy` | 回 ACK → ProcessManager 根据 driver_type 获取 Driver → Driver.deploy(InstanceDeployConfig) → 上报 `agent.instance_started` |
| `instance.terminate` | 回 ACK → ProcessManager 调用 Driver.terminate() → 清理资源 → 上报 `agent.instance_stopped` |
| `task.assign` | 回 ACK → MessageRouter 找到 instance_id 对应的 Agent（必须已 RUNNING）→ Driver.send_task() |
| `task.cancel` | 回 ACK → ProcessManager 向 Agent 发 SIGTERM → 上报 `agent.task_cancelled` |
| `node.heartbeat_ack` | 更新最后一次心跳响应时间 |
| `node.token_refresh_required` | 调 REST 换发 Token，重建连接 |
| `node.config_update` | 回 ACK → ConfigManager 更新内存配置 |
| `node.upgrade_required` | 记录日志 → 优雅停止所有实例 → 退出进程（由系统服务管理器拉起新版本） |
| `sync.snapshot` | 回 ACK → reconciliation：对比内存中的实例状态与 snapshot，terminate 多余实例、deploy 缺失实例 |

#### 3.1.3 ACP 端上报消息

| 上报消息 | 触发条件 | 内容 |
|---------|---------|------|
| `node.heartbeat` | 定时器（默认 30s） | `{ node_id, timestamp, active_instances: 2, mem_usage_mb: 128 }` |
| `agent.instance_started` | 实例部署成功（Agent 子进程启动或 HTTP 连接建立） | `{ instance_id, prototype_id, driver_type, pid? }` |
| `agent.instance_stopped` | 实例终止（主动 terminate 或异常退出） | `{ instance_id, prototype_id, exit_code?, reason }` |
| `agent.task_progress` | Agent 流式输出（逐 chunk） | `{ instance_id, content, is_final: false, client_msg_id }` |
| `agent.task_completed` | Agent 正常完成任务 | `{ instance_id, result, evidence, duration_secs, client_msg_id }` |
| `agent.task_failed` | Agent 异常退出或执行失败 | `{ instance_id, reason, exit_code?, client_msg_id }` |
| `agent.task_cancelled` | Agent 收到取消信号后退出 | `{ instance_id, client_msg_id }` |

### 3.2 节点 ↔ Agent 通信

#### 3.2.1 通信模型

节点 与 Agent 之间的通信基于 **Driver 模式**：每种 Agent 类型对应一个 Driver 实现，统一抽象为 `AgentDriver` trait。节点 不关心 Agent 的内部实现，只通过 Driver trait 与 Agent 交互。

**消息分层**：实例生命周期（deploy/terminate）和任务执行（assign/cancel）是两个独立层次——先有实例，再向实例派发任务。一个实例可接收多个任务。

```
Server JSON-RPC          ACP 协议适配层              Agent 原生接口
─────────────         ─────────────────          ──────────────

实例生命周期层：
instance.deploy ──→ ProtocolAdapter ───→ HermesDriver ──────→ hermes --instance-id ... --system-prompt ... --stream-output
                    (解析 InstanceDeployConfig)  (CLI spawn)

instance.deploy ──→ ProtocolAdapter ───→ ClaudeCodeDriver ──→ claude --system-prompt ... --allowedTools ... --output-format stream-json
                    (解析 InstanceDeployConfig)  (CLI spawn)

instance.deploy ──→ ProtocolAdapter ───→ HttpApiDriver ─────→ POST /api/v1/init { config }
                    (解析 InstanceDeployConfig)  (HTTP init)

instance.terminate → ProtocolAdapter ───→ 任意 Driver ──────→ SIGTERM / 关闭连接

任务执行层：
task.assign ──────→ ProtocolAdapter ───→ HermesDriver ──────→ stdin: { "task_spec": ... }
                    (JSON-RPC → Agent格式)       (stdin 管道)

task.assign ──────→ ProtocolAdapter ───→ ClaudeCodeDriver ──→ stdin: { "prompt": "..." }
                    (JSON-RPC → Agent格式)       (stdin 管道)

task.assign ──────→ ProtocolAdapter ───→ HttpApiDriver ─────→ POST /api/v1/execute { task, context }
                    (JSON-RPC → Agent格式)       (HTTP + SSE)
```

#### 3.2.2 HermesDriver

Hermes 走 CLI 子进程模式。

**部署（instance.deploy）**：
```bash
hermes --instance-id <instance_id> \
       --system-prompt "<agent_config.system_prompt>" \
       --model <agent_config.model> \
       --tools <agent_config.tools> \
       --stream-output \
       --context-dir <working_dir>
```

`instance.deploy` 的 `InstanceDeployConfig` 翻译为 CLI 参数：
| 配置字段 | CLI 参数 | 说明 |
|---------|---------|------|
| `agent_config.system_prompt` | `--system-prompt` | 原型定义的角色提示 |
| `agent_config.model` | `--model` | 使用的模型 |
| `agent_config.tools` | `--tools` | 允许的工具列表 |
| `agent_config.temperature` | `--temperature` | 温度参数 |
| `executable.path` | 可执行文件路径 | 如 `/usr/local/bin/hermes` |
| `working_dir` | `--context-dir` | 工作目录 |

**任务分配（task.assign）**：通过 stdin 管道传入任务上下文 JSON

**输出**：
- stdout 逐行输出 JSON 流：`{"type": "progress", "content": "..."}` 或 `{"type": "result", "content": "...", "evidence": [...]}`
- stderr 输出日志（ACP 仅记录到本地日志，不转发 Server）

**退出码**：0 = 正常完成，非0 = 执行失败

**流式转发映射**：

| Hermes stdout | ACP 转发 |
|---------------|---------|
| `{"type": "progress", "content": "..."}` | `agent.task_progress { content, is_final: false }` |
| `{"type": "result", "content": "...", "evidence": [...]}` | `agent.task_completed { result, evidence }` |
| `{"type": "error", "message": "..."}` | `agent.task_failed { reason: message }` |

#### 3.2.3 ClaudeCodeDriver

Claude Code 是 Anthropic 的 CLI Agent，走 CLI 子进程模式。

**部署（instance.deploy）**：
```bash
claude --system-prompt "<agent_config.system_prompt>" \
       --model <agent_config.model> \
       --output-format stream-json \
       --allowedTools <agent_config.tools> \
       --working-dir <working_dir>
```

`instance.deploy` 的 `InstanceDeployConfig` 翻译为 CLI 参数：
| 配置字段 | CLI 参数 | 说明 |
|---------|---------|------|
| `agent_config.system_prompt` | `--system-prompt` | 原型定义的角色提示 |
| `agent_config.model` | `--model` | 使用的模型（如 `claude-sonnet-4-20250514`） |
| `agent_config.tools` | `--allowedTools` | 允许的工具列表（如 `Read,Grep,Bash`） |
| `executable.path` | 可执行文件路径 | 如 `claude`（$PATH 查找） |
| `working_dir` | `--working-dir` | 工作目录 |

**任务分配（task.assign）**：通过 stdin 管道传入任务描述

**输出**：
- stdout 逐行输出 stream-json 格式：`{"type": "assistant", "content": [...]}` 流式增量
- 退出码：0 = 正常完成

**协议翻译要点**：
- Claude Code 的 stream-json 输出需要提取 `content` 字段中的文本内容，映射为 `agent.task_progress`
- Claude Code 的工具调用结果（如文件写入）需要提取为 evidence
- Claude Code 不支持自定义 evidence 格式，ACP 需要从退出后的工作目录 diff 中提取变更文件列表作为 evidence

**流式转发映射**：

| Claude Code stdout | ACP 转发 |
|-------------------|---------|
| `{"type": "assistant", "content": [{"type": "text", "text": "..."}]}` | `agent.task_progress { content: text, is_final: false }` |
| `{"type": "result", ...}` | `agent.task_completed { result, evidence: [工作目录 diff] }` |
| 退出码 0 | 触发终态消息发送 |

#### 3.5.2 生命周期与任务的关系

**关键设计**：实例生命周期（deploy/terminate）和任务执行（assign/cancel）是**两个独立层次**。先有实例，再向实例派发任务。

```
实例层：  instance.deploy ──→ RUNNING ──→ instance.terminate ──→ STOPPED

任务层：                     task.assign ──→ Agent 执行 ──→ task_completed
                            （可多次，实例在 RUNNING 期间可接收多个任务）
```

**为什么分开**：

| 问题 | 如果 deploy 和 assign 合一 | 分开后 |
|------|--------------------------|--------|
| 一个"代码审查员"连续审查 3 个文件 | 每次都重新 spawn Agent（Claude Code 冷启 3-5s） | spawn 一次，3 个任务复用实例 |
| 实例空闲 10 分钟 | Server 无法回收，只能等 Agent 自己退出 | Server 主动 `instance.terminate` 回收资源 |
| Agent 崩溃 | 混淆"实例挂了"和"任务挂了" | 实例崩溃 → `instance_stopped`；任务失败 → `task_failed`，语义清晰 |

#### 3.5.3 Server 端的职责（ACP 视角）

ACP 不关心 Server 内部如何管理原型和实例，但需要理解 Server 下发的消息含义：

| Server 行为 | 触发条件 | ACP 收到的消息 |
|------------|---------|---------------|
| 从原型实例化 | 复合型研发经理创建数字员工实例 | `instance.deploy`（含已解析的 `InstanceDeployConfig`） |
| 分配任务 | 任务调度器将任务分给已运行实例 | `task.assign`（需实例已 RUNNING） |
| 回收空闲实例 | 实例空闲超时 / 项目结束 | `instance.terminate` |
| 崩溃后重建 | ACP 上报 `instance_stopped`，Server 决定重试 | 新的 `instance.deploy`（同一 prototype_id，新 instance_id） |
| 全量同步 | ACP 重连后 snapshot 对账 | `sync.snapshot`（包含期望的实例列表） |

**原型解析发生的位置**：Server 的 `agent-svc` 将 `AgentPrototype`（原型定义）+ `RuntimeConfig`（运行配置）解析为 `InstanceDeployConfig`（操作层参数）后下发给 ACP。ACP 不做原型业务语义解析，只消费操作层配置。

```
agent-svc（Server 端）                     ACP（终端端）
───────────────────                      ───────────────

AgentPrototype                           InstanceDeployConfig
  ├─ name: "代码审查员"                     ├─ driver_type: "claude-code"
  ├─ capabilities: [...]                   ├─ executable: { path, args }
  └─ runtime_config: { ... }        ──→   ├─ agent_config: { system_prompt, model, tools }
                                           ├─ credentials: { API_KEY: "..." }
                                           └─ resource_limits: { ... }

解析过程：Server 从原型的业务定义
提取出 ACP 可直接消费的操作参数
ACP 不需要理解"代码审查员"意味着什么
```

### 3.6 密钥管理

#### 3.6.1 MVP 阶段：Server 直接下发临时凭证

```
Server（agent-svc）                    ACP                          Agent 子进程
─────────────────                    ───                         ────────────

1. 生成临时凭证（有效期有限）
   credentials: {
     ANTHROPIC_API_KEY: "sk-ant-..."
   }
                  ── instance.deploy ──→
                                        2. 提取 credentials
                                        3. 注入到 Agent 子进程
                                           环境变量中
                                                                     env: ANTHROPIC_API_KEY=sk-ant-...
                                                                     Agent 启动后直接使用
```

**MVP 策略说明**：
- Server 的 `agent-svc` 在处理 `instance.deploy` 时，从密钥存储（环境变量/配置中心）读取明文凭证，放入 `credentials` 字段
- ACP 收到后直接注入到 Agent 子进程的环境变量中，自身不存储
- 临时凭证由 Server 端控制有效期（如 24 小时），过期后 Agent 调用会报鉴权失败，触发 `task_failed`
- **安全边界**：credentials 在 WebSocket 连接内传输（TLS 加密），ACP 进程内存中短暂存在，Agent 子进程启动后 ACP 释放引用

**MVP 的局限**：
- 凭证明文经过 ACP 内存（虽然短暂）
- 无法动态轮换凭证（需要 terminate → 重新 deploy）
- 无法精细控制凭证权限范围

#### 3.6.2 v2 演进：Vault 引用模式

```
Server（agent-svc）                    ACP                          密钥管理服务
─────────────────                    ───                         ────────────

1. 下发引用而非明文
   credentials: {
     ANTHROPIC_API_KEY: "vault://secrets/anthropic-key"
   }
                  ── instance.deploy ──→
                                        2. 解析 vault:// 引用
                                                           ── 3. 请求凭证 ──→
                                                           ←── 4. 返回明文 ───
                                        5. 注入 Agent 环境变量
                                        6. 释放明文引用
```

**v2 改进**：
- ACP 从 `vault://` 引用解析为明文的过程由 ACP 的 `SecretManager` 模块完成
- 支持凭证动态轮换：Server 下发 `instance.credential_rotate` 消息，ACP 重新解析引用并热更新 Agent 环境变量
- 支持凭证权限范围控制（如只允许访问特定模型 API）
- Server 不再接触凭证明文，安全边界更清晰

## 4. 跨平台策略

### 4.1 编译目标

| 阶段 | 目标平台 | target triple | 优先级 |
|------|---------|---------------|--------|
| MVP | macOS (Apple Silicon) | `aarch64-apple-darwin` | P0 |
| MVP | macOS (Intel) | `x86_64-apple-darwin` | P0 |
| MVP | Linux (x86_64) | `x86_64-unknown-linux-gnu` | P0 |
| 成长期 | Linux (ARM64) | `aarch64-unknown-linux-gnu` | P1 |
| 成长期 | Windows | `x86_64-pc-windows-msvc` | P2 |
| 成熟期 | Android | `aarch64-linux-android` | P3 |
| 成熟期 | iOS | `aarch64-apple-ios` | P3 |

### 4.2 平台差异处理

通过 Rust 的 `cfg` 条件编译和 trait 抽象隔离平台差异：

| 差异点 | 抽象方式 | 说明 |
|--------|---------|------|
| 进程管理 | `ProcessManager` trait | Unix 用 `fork/exec` + `signal`；Windows 用 `CreateProcess` + `TerminateProcess`；移动端不做子进程管理（走 HTTP Agent） |
| 文件系统 | `FileSystem` trait（或直接用 `dirs` crate） | 配置文件路径：Unix `$HOME/.legionos/`，Windows `%APPDATA%\LegionOS\`，移动端应用沙箱 |
| 网络 | tokio 统一异步 IO | WebSocket/HTTP 全部走 tokio，无需平台适配 |
| 系统服务 | `ServiceManager` trait | macOS: `launchd` plist；Linux: systemd unit；Windows: Windows Service；移动端：前台 Service / Background Task |
| 信号处理 | `signal_hook` crate (Unix) | SIGTERM/SIGINT 优雅关闭；Windows 用 `SetConsoleCtrlHandler` |

**移动端特殊处理**：

Android / iOS 不支持 spawn 子进程，ACP 在移动端降级为纯 HTTP Agent 模式——只使用 HttpApiDriver，不启动 CLI 子进程。代码通过 `cfg(target_os)` 在编译期排除 ProcessManager 的子进程逻辑。

## 5. 安装与升级

### 5.1 安装方式

| 平台 | 安装方式 | 命令示例 |
|------|---------|---------|
| macOS | Homebrew | `brew install legionos/tap/acp` |
| Linux | Shell 脚本 | `curl -fsSL https://legionos.example.com/install.sh \| sh` |
| Windows | MSI 安装包 | 下载 MSI，双击安装 |
| 通用 | 单二进制下载 | 从 GitHub Release 下载对应平台二进制，放入 PATH |

**安装后自动操作**：
1. 生成 `device_code`（UUID v4），写入 `$HOME/.legionos/device_code`
2. 注册系统服务（macOS: launchd，Linux: systemd），设置开机自启
3. 首次启动后进入"待激活"状态，等待用户在 OS 后台获取 `activation_code`

**激活流程**：
```bash
# 方式一：CLI 激活
acp activate --code <activation_code>

# 方式二：交互式激活（提示输入 activation_code）
acp activate
```

### 5.2 升级机制

**升级检测**：

- ACP 启动时和运行中（每 6 小时）检查 Server 是否推送 `node.upgrade_required` 消息
- 同时定期查询 `GET /api/agent-node/check-update?current_version=<version>`

**升级策略**：

| 策略 | 说明 |
|------|------|
| 被动升级 | Server 推送 `node.upgrade_required`，ACP 下载新版本并重启 |
| 主动升级 | 用户执行 `acp upgrade`，检查并安装最新版本 |
| 灰度升级 | Server 可按 node_id 维度控制升级节奏，避免全量同时升级 |

**升级流程**：

```
1. ACP 收到升级通知（或主动检测到新版本）
2. 下载新版本二进制到临时目录
3. 校验 SHA256 签名
4. 优雅停止所有 Agent 子进程（SIGTERM → 10s → SIGKILL）
5. 上报 agent.instance_stopped 给 Server
6. 替换旧二进制（原子替换：rename 新文件到旧路径）
7. 重启 ACP 进程（由系统服务管理器拉起）
8. 新版本 ACP 重连 Server → sync.snapshot → 恢复执行
```

**版本回退**：

- 升级前保留旧版本二进制到 `$HOME/.legionos/acp.bak`
- 新版本启动失败（连续 3 次连接 Server 失败或启动后立即崩溃）→ 自动回退到旧版本
- 手动回退：`acp rollback`

## 6. 配置管理

### 6.1 配置文件格式

配置文件路径：`$HOME/.legionos/acp.toml`

```toml
[server]
url = "wss://legionos.example.com/agent-ws"
rest_url = "https://legionos.example.com"

[connection]
heartbeat_interval_secs = 30
reconnect_max_backoff_secs = 60
reconnect_max_retries = 0   # 0 = 无限重试

[logging]
level = "info"              # trace | debug | info | warn | error
file_path = "$HOME/.legionos/acp.log"
max_file_size_mb = 50
max_file_count = 5

[[agents]]
type = "hermes"
path = "/usr/local/bin/hermes"
default = true

[[agents]]
type = "claude-code"
path = "claude"             # $PATH 中的 claude 命令

[[agents]]
type = "http-api"
default_endpoint = "http://localhost:8090"
```

### 6.2 配置优先级

优先级从高到低：

1. **CLI 参数**：`acp --server-url wss://...`
2. **Server 运行时覆盖**：`node.config_update` 消息（仅存于内存，不写回文件）
3. **环境变量**：`LEGIONOS_SERVER_URL`、`LEGIONOS_LOG_LEVEL`
4. **配置文件**：`acp.toml`
5. **编译期默认值**

### 6.3 Server 端配置同步

ACP 连接 Server 后，Server 可通过以下方式影响 ACP 运行时行为：

| Server 消息 | 影响范围 | 持久化 |
|------------|---------|--------|
| `node.config_update` | 心跳间隔、重连策略等运行时参数 | 不持久化（重启后恢复为本地配置） |
| `instance.deploy` 中的 `InstanceDeployConfig` | 单个 Agent 实例的部署配置（Driver、启动参数、agent_config、credentials） | 不持久化（ACP 不落盘任何实例配置） |
| `instance.terminate` | 单个 Agent 实例的终止和资源回收 | 不持久化 |
| `node.upgrade_required` | 触发 ACP 版本升级 | 不持久化 |

**设计理由**：ACP 坚持"本地配置为权威源 + Server 运行时覆盖"的策略。Server 覆盖不落盘，确保 ACP 重启后行为一致且可预测，避免"配置漂移"问题。
