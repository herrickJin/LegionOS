## ADR-XXX: [决策标题]

### 状态
Accepted

### 上下文
pdm-svc（原 core-svc）聚合了经营决策、项目管理、项目实施三个域，数据特征是**结构化主体 + 动态扩展字段**混合：

- 结构化主体：商机标题/状态/创建人、项目名称/经理、任务标题/指派人 → 列固定，需要关系型查询和事务
- 动态扩展字段：商机评估结果（结构不固定）、任务元数据（不同任务类型字段不同）、资源技能标签（数组/嵌套）→ 结构迭代，需要灵活存储和按内容查询

原始方案是 MySQL 存结构化 + MongoDB 存动态字段，但引入两个数据库实例增加了运维复杂度，且 MVP 阶段团队小、数据量低，维护两个数据库进程是净负担。

核心问题：**能否用单一数据库同时满足结构化查询和动态字段索引查询？**

### 选项

- **选项A：MySQL + JSON 列** — 团队可能更熟悉 MySQL；JSON 类型可存储动态字段；但 JSON 列查询需预先为每个路径建虚拟生成列+索引，无法对未预定义路径建索引；每次查询解析 JSON 字符串，无索引时全表扫描
- **选项B：PostgreSQL + JSONB** — JSONB 二进制存储，查询无需重新解析；GIN 索引一次建立即可支持任意路径查询，无需预定义虚拟列；`@>` 包含查询语法直接命中索引；额外收益：LISTEN/NOTIFY 通知、部分索引、行级安全等能力
- **选项C：MySQL + MongoDB 双库** — 各取所长，MongoDB 原生文档查询能力强；但多一个数据库实例意味着多一套备份/监控/升级/连接池运维；跨库数据一致性需分布式事务或手动补偿

### 决策
选择 **PostgreSQL + JSONB 统一存储**。

理由：pdm-svc 的动态字段（评估结果、任务元数据、资源技能标签）不是"存了就完事"的存档字段，而是**业务查询维度**——需要按风险等级过滤商机、按技能标签匹配资源。PostgreSQL 的 GIN 索引让"动态结构 + 高效查询"成为可能，省掉了 MySQL 方案中预测字段和建虚拟列的维护成本，也省掉了 MongoDB 方案的运维开销。

### 后果

- **好处**：单一数据库实例，运维复杂度减半；动态字段查询无需预定义索引路径；JSONB 与关系型查询可同一事务中完成
- **代价**：团队需熟悉 PostgreSQL 特有语法（GIN 索引、`@>`/`?` 操作符、JSONB 函数）；JSONB 写入比纯列存储略慢（需解析为二进制），但对 pdm-svc 的写频次可忽略
- **约束**：动态字段单条不超过 1MB（PostgreSQL 行限制 8KB 页，TOAST 自动外置压缩）；如未来某域的动态字段超过此限制需评估拆分
- **风险**：如果团队完全没有 PostgreSQL 经验，初期有学习曲线；缓解：JSONB 查询模式在 schema.sql 中有示例，且 Hibernate Types 封装了映射细节

## ADR-002: 核心业务服务命名 — pdm-svc

### 状态
Accepted

### 上下文
MVP 阶段将经营决策、项目管理、项目实施三个高频联动域聚合为一个服务，原命名 `core-svc` 过于宽泛——"core"不传达任何业务含义，无法从名字推断服务职责，且容易让人误以为它是"基础设施核心"而非"业务聚合核心"。

核心问题：**如何用服务名直接传达"商机→项目→任务→交付"的产研交付链路职责？**

### 选项

- **选项A：delivery-svc** — 突出"交付"主线，但语义偏终端，容易让人以为只管交付阶段
- **选项B：pdm-svc（Product Delivery Management）** — 项目交付管理，显式体现项目+交付的组合含义，缩写简短，口语化方便
- **选项C：bizops-svc** — 业务运营服务，强调业务运转；但"运营"在中文语境容易跟用户运营/内容运营混淆

### 决策
选择 **pdm-svc（Product Delivery Management）**。

理由：pdm 准确表达了"从商机到交付的产研交付管理"这一核心职责，缩写简短、读音清晰，且与后续拆分方向（`decision-svc`/`project-svc`/`execution-svc`）形成清晰的命名体系——pdm 是这三者的上位概念。

### 后果

- **好处**：服务名即业务含义，新人一看就知职责范围；拆分后命名体系自洽（pdm → decision/project/execution）
- **代价**：pdm 不是行业通用缩写，首次接触需解释全称；团队沟通时需约定读法（P-D-M 三个字母还是连读）
- **约束**：Java 包名为 `com.legionos.pdm`，后续拆分时子服务包名需保持一致规范（如 `com.legionos.decision`）

## ADR-003: 技术网关选型 — Spring Cloud Gateway

### 状态
Accepted

### 上下文
OS 服务端采用微服务架构，外部流量（Web 前端、移动端、CLI、第三方集成）需要统一入口进行路由、认证、限流、熔断等处理。当前架构中 `portal-svc` 作为 BFF 层承担了部分入口职责，但缺少独立的技术网关层，导致：
- JWT 验证逻辑散落在各服务中，重复实现
- 无法统一限流，单服务过载可能拖垮整个系统
- WebSocket 连接（ACP 通信）缺乏入口级管理
- 缺少全链路追踪头注入，排障困难

核心问题：**如何选择技术网关方案，在 MVP 阶段快速落地，同时支持成长期和成熟期的演进？**

### 选项

- **选项A：Spring Cloud Gateway** — 与 Spring 生态无缝集成，支持 WebSocket 路由，Resilience4j 熔断开箱即用；但基于 WebFlux（响应式），学习曲线较陡，相比 Nginx 资源消耗更高
- **选项B：Nginx + Lua（OpenResty）** — 性能极高（C 层），配置简单，WebSocket 原生支持；但 Lua 开发效率低，与 Java 生态集成需额外工作
- **选项C：Kong** — 插件生态丰富（认证/限流/日志均有现成插件），支持 gRPC；但商业版功能收费，插件调试较复杂
- **选项D：Istio Ingress Gateway** — 与 K8s 深度集成，支持 mTLS、流量管理、可观测性；但重量级，MVP 阶段引入过早

### 决策
MVP 阶段使用 **Spring Cloud Gateway**。

理由：
1. 与后端服务同技术栈（Java/Spring Boot），团队无需学习新语言
2. 原生支持 WebSocket 路由（数字员工 ACP 通信必需）
3. 集成 Resilience4j 实现熔断降级，与各服务的熔断配置一致
4. MVP 阶段流量低，JVM 额外开销可接受
5. 成熟期可平滑迁移至 Istio Ingress Gateway

### 后果

- **好处**：全 Java 技术栈，开发调试效率高；WebSocket 路由无需额外配置；与 Resilience4j/Redis 生态集成自然
- **代价**：比 Nginx 多一个 JVM 进程；WebFlux 响应式编程模型对团队有学习曲线；MVP 单实例部署无高可用
- **约束**：MVP 阶段可用 Nginx 反向代理做简化替代；成长期需多实例 + 负载均衡；成熟期评估迁移 Istio
- **风险**：如果团队完全没有响应式编程经验，初期开发可能较慢；缓解：网关逻辑尽量简单（只做路由+认证+限流），复杂业务逻辑不下沉

## ADR-004: WebSocket/ACP 通信和 AI 能力调用归属 inf-svc

### 状态
Accepted

### 上下文
原始设计中 `agent-svc` 聚合了三类不同性质的职责：
1. 数字员工业务逻辑（原型管理、实例生命周期、任务分配）
2. WebSocket/ACP 通信（长连接管理、心跳、消息中转）
3. AI 能力调用（LLM 路由、Token 计量、成本追踪）

这三类职责的变化原因不同：业务逻辑随需求变化，通信层随协议/连接数变化，AI 能力随模型供应商变化。放在同一服务中违反了单一职责原则，且 WebSocket 连接管理和 LLM 调用可能被其他域（knowledge-svc、evolution-svc）复用。

核心问题：**WebSocket 通信和 AI 能力调用应该归属哪个服务？**

### 选项

- **选项A：留在 agent-svc** — 当前最简单；但多域复用时需跨服务调用 agent-svc，违反域独立性
- **选项B：归属 inf-svc** — inf-svc 已有 Body 运行时管理和模型管理，WebSocket 通信和 AI 调用是其自然延伸，形成完整的"数字员工运行时基础设施"闭环
- **选项C：各自独立新服务** — WebSocket 独立为 relay-svc，AI 调用独立为 ai-platform-svc；但 MVP 阶段服务数量过多

### 决策
**WebSocket/ACP 通信和 AI 能力调用统一归属 `inf-svc`**。

理由：
1. inf-svc 已有 Body 运行时管理，WebSocket 是运行时的通信层，放一起形成"创建 Body → 建立连接 → 推送任务 → 接收结果"的完整闭环
2. AI 能力调用与模型管理天然关联（模型配置 → LLM 路由 → Token 计量 → 成本追踪）
3. MVP 阶段避免引入新服务，减少运维负担
4. agent-svc 专注业务逻辑（"人事部"），inf-svc 专注运行时（"机房"），职责清晰

### 后果

- **好处**：agent-svc 职责单一，只管业务逻辑；inf-svc 形成完整的运行时闭环；其他域（knowledge-svc、evolution-svc）可直接调用 inf-svc 的 AI 能力
- **代价**：inf-svc 变重，需关注资源隔离（WebSocket 连接占用内存、LLM 调用占用线程）；agent-svc 需通过 Feign 间接调用，多一跳延迟
- **约束**：agent-svc 通过 Feign 客户端调用 inf-svc 的 WebSocket 推送接口和 LLM 生成接口；inf-svc 接收 ACP 结果后通过事件/回调通知 agent-svc
- **风险**：inf-svc 职责过重可能成为瓶颈；缓解：成长期可将 AI 能力调用拆分为独立的 ai-platform-svc
