# 一级产研流程规范

> **定位**：产研流程OS的核心流程规范，定义"少了就会系统性地出错、多了就能系统性地变好"的规范集合
> **状态**：初稿
> **关联文档**：01-业务架构设计（产研流程域）、PRD 第4章（任务规范6要素）

---

## 1. 一级规范的判断标准

不是看"重不重要"——12个域都重要。一级的判断标准是：

> **如果这个规范变了，下面所有东西都要跟着变。** 它处于依赖链的最顶端，blast radius 最大。

一个规范被定为一级，意味着：
- 它被其他所有规范/任务/流程引用
- 它的定义决定了其他规范的行为边界
- 改了它等于改了整个系统的行为契约

## 2. 七个一级规范

### 2.1 任务规范 6 要素定义体系 ✅ 已有

**失控时刻：** 人说"把这事做了"，做完后发现理解的"做完"不一样。没有统一的规则定义、步骤拆解、验收标准、执行者责任——每个人按自己的习惯来。

**为什么是一级：**
OS 的核心抽象本身。15 个典型任务规范全依赖 6 要素；项目规范是它的变体；子任务嵌套机制依赖它；判定规则引擎依赖第 5 要素"执行岗责任要求"；工作台依赖第 6 要素。如果 6 要素结构改了，一切重来。

**包含内容（已在 PRD 第4章定义）：**
- 任务元数据（名称/类型/适用场景/优先级/预估工时）
- 输入及输入标准
- 核心产出物及验收标准
- 子步骤任务规范 + 任务流
- 执行岗责任要求
- 执行工作台（工具集、常用操作、工作区布局、外部资源）

### 2.2 需求结构化与准入规范

**失控时刻：** 需求已经拆成任务、开发已经做了一周，产品方过来说"我说的不是这个意思"。回头一看，原始需求就是一句话，没有结构化的记录，谁也说不清"到底谁理解错了"。

**为什么是一级：**
它是 T9 "需求层设计"的前置步骤。T9 的假设是"需求已经清楚，我们来设计怎么实现"——但这个假设在现实中经常不成立。需求从"模糊的一句话"到"可以进入设计阶段的结构化需求"，中间有一个断层，不填上，设计做得再好也是错的。

**包含内容：**
- 需求的显式化（把一句话变成一个结构化的需求描述）
- 不可协商条件的识别（"这个必须支持" vs "这个可以讨论"）
- 歧义消除的 checklist（边界条件、异常流程、状态变化——这些在口述需求时永远被忽略）
- 需求可行性的快速判定（技术可行 & 资源可行，而不是等到架构设计阶段才发现做不了）
- 需求准入标准（满足什么条件才进入 T9 设计阶段，不满足就退回）

**与现有规范的关系：** T9 的前置步骤，详细规范见 [T16-需求结构化与准入规范](./T16-需求结构化与准入规范.md)。

### 2.3 技术方案评审规范

**失控时刻：** 3 个月后新同事问"这块为什么用 Kafka 而不是 RabbitMQ"，没人记得。6 个月后 Kafka 出问题了想换，决策链断了——不知道当时的约束条件是什么，不敢换。

**为什么是一级：**
T10-T14 覆盖了"做设计"，但完全没有"审设计"。一个架构决策做错了，后面 6 个月的开发都在为这个错误买单。技术方案评审不是 Code Review 的宏观版，它是一个独立的决策质量保障流程——判断的不是代码写得对不对，而是方向选得对不对。

**包含内容：**
- 评审触发条件（什么规模的设计必须走评审，什么可以自决）
- 评审参与者规则（架构师必须参加 / 相关模块 owner 必须参加 / 新人设计的方案必须走评审）
- 评审 checklist（技术选型是否有 ADR / 是否有备选方案对比 / 是否考虑了失败场景 / 是否评估了引入的技术债务）
- 评审结论类型（通过 / 有条件通过 + 待办项 / 驳回 + 重做方向）
- 架构决策记录（ADR）的强制产出——不只是"评审过了"，而是"当时为什么选这个方案"被记录下来

**与现有规范的关系：** T10-T14 的下游保障步骤，每个设计类任务完成后触发。当前完全缺失。

### 2.4 发布与回滚规范

**失控时刻：** 周五下午 5 点发布，6 点发现用户登录全挂了。想回滚，发现数据库迁移脚本没有回滚方案。周末全员加班手动修数据。

**为什么是一级：**
产研流程的"最后一公里"。代码写完了、测试通过了、Review 过了——然后怎么上线？15 个任务规范没有一个涉及发布。而线上事故的根因统计里，发布环节的事故占比极高。不是代码没写好，是发布流程没有规范。

**包含内容：**
- 发布策略选择（全量 / 灰度 / 蓝绿 / 金丝雀——什么场景选什么策略）
- 发布 checklist（依赖服务是否就绪 / 数据库迁移是否先于应用部署 / 配置项是否在所有环境验证 / 回滚脚本是否就绪）
- 发布时间窗规则（哪些时段允许发布 / 哪些时段禁止 / 紧急修复的例外流程）
- 回滚触发条件（什么信号出现必须回滚——错误率超过 X% / P99 延迟超过 Yms / 核心业务指标下降 Z%）
- 回滚后复盘要求（回滚不是终点，为什么没在灰度阶段发现才是）

**与现有规范的关系：** 完全没有对应的 T 任务，最大的空白区域。

### 2.5 缺陷管理规范

**失控时刻：** 测试提了 20 个 Bug，开发修了 15 个说"修完了"，测试回归发现 5 个没修、3 个修出了新 Bug。没有人能说清楚当前版本到底还有多少未解决的缺陷、哪些影响发布。

**为什么是一级：**
T6（测试用例开发）和 T8（测试执行）覆盖了"发现缺陷"，但没有覆盖"缺陷被发现之后怎么办"。缺陷的生命周期管理是一个独立的流程，而且它直接关系到——这个版本能不能发、这个迭代的完成度到底是多少。

**包含内容：**
- 缺陷分级标准（P0 阻塞发布 / P1 影响核心功能 / P2 有 workaround / P3 体验问题）
- 缺陷定责与分配规则（不是追责，而是——谁最有能力修？修的优先级相对其他任务怎么排？）
- 缺陷修复的验证闭环（修了 → 回归测试通过 → 关联用例更新 → 关闭）
- 缺陷趋势分析（这个模块是不是缺陷密度在上升？是不是某个人的代码缺陷率偏高？——用于驱动流程改进，不是用于考核）
- 缺陷与需求的关联追溯（这个缺陷是从哪个需求的哪个验收条件漏掉的？→ 反馈给需求结构化规范 2.2）

**与现有规范的关系：** T6/T8 的下游管理流程，当前缺失。

---

### 2.6 接口契约管理规范

**失控时刻：** A 团队重构了一个接口，"放心，向下兼容的"。上线后 B 团队的服务开始报 500，排查两小时发现 A 团队把字段类型从 string 改成了 enum，旧客户端传的字符串值不在 enum 里。A 团队说"我们不知道你们在用那个字段"。

**为什么是一级：**
多团队协作的核心痛点。T11 概要设计提到了"接口设计"，但缺的是"接口契约的版本管理和变更通知"。一个字段改动，导致三个团队各浪费半天排查——这个问题随着团队规模增长指数级恶化。

**包含内容：**
- 接口契约的定义格式（OpenAPI / Protobuf——统一一种，不允许多种并存）
- 接口版本策略（向后兼容变更怎么做 / 破坏性变更怎么做 / 废弃字段的标记和删除周期）
- 变更通知机制（接口变更 → 自动通知所有消费者 → 消费者确认兼容性 → 设定废弃时间线）
- 接口契约的消费方注册（谁在用这个接口？不知道谁在用就不敢改）
- 契约测试（Contract Testing）——不是"我改了之后跑了自己的测试"，而是"我的改动是否破坏了你的预期"

**与现有规范的关系：** T11 接口设计的版本化与治理延伸，当前缺失。

---

### 2.7 故障应急响应规范

**失控时刻：** 线上挂了 40 分钟，群里 200 条消息，5 个人同时在操作、3 个人在问"现在什么情况"、没人知道谁在做决策。恢复后没人写复盘，1 个月后同样的问题再次发生。

**为什么是一级：**
发布管理规范（2.4）管的是"怎么不出事"，故障应急规范管的是"出事了怎么办"。这是产研流程里压力最大、时间最紧、信息最乱的环节。"运维在喊、产品在催、CEO 在群里问"——没有流程规范的情况下，响应全凭个人经验和肾上腺素。

**包含内容：**
- 故障分级标准（P0 全站不可用 / P1 核心功能不可用 / P2 部分用户受影响 / P3 体验降级）
- 应急响应角色与职责（故障指挥官 / 处理人 / 沟通人——不是谁技术强谁上，是角色明确）
- 升级路径（P0 15 分钟内没定位到 → 升级到谁；P1 1 小时内没恢复 → 升级到谁）
- 沟通规范（什么时候通知用户 / 通知什么内容 / 什么时候更新状态页 / 谁来发）
- 故障复盘规范（事后必须产出：Timeline / 根因分析 / 改进措施 / 改进验证时间）
- 改进措施的闭环跟踪（复盘提了 5 个改进项，3 个月后还有 3 个没做——这本身就是一个信号）

**与现有规范的关系：** 完全不依赖其他规范，独立的应急领域。当前完全缺失。

## 3. 七个规范的关系

```
需求结构化 ──→ 技术方案设计(T9-T14) ──→ 方案评审 ──→ 开发执行(T4/T5) ──→ Code Review(T7)
     ↑                                                                              ↓
     │                                                                        测试(T6/T8)
     │                                                                              ↓
     │                                                                        缺陷管理
     │                                                                              ↓
     └──────── 故障复盘 ←──── 故障应急 ←──── 线上故障 ←──── 发布与回滚 ←───────────┘

任务规范 6 要素定义体系 ──→ 贯穿全流程（所有 T 任务的定义基础）
接口契约管理 ──→ 贯穿全流程（设计阶段定义、开发阶段遵守、发布阶段验证）
```

## 4. T2/T3 拆分重构

### 原定义（PRD 第7章）

- T2：Old Java 项目梳理
- T3：Old Web 项目梳理

按技术栈（语言）拆分。

### 问题

分析维度取决于项目领域（前端/后端/微服务/数据管道），而非语言。同一语言的不同架构形态，分析工具链和关注点完全不同。

### 重构后

**T2 遗留前端项目分析**：Web / 小程序 / App 共用

- 输入：代码仓库 + 项目形态参数
- 分析维度（固定）：
  - 组件树与复用度——公共组件、一次性组件、嵌套层级、循环引用
  - 路由与页面结构——页面数量、路由层级、权限控制点位置
  - 状态管理拓扑——全局/局部状态划分、冗余状态、触发链路
  - 接口调用全景——调用清单、参数硬编码、异常态处理覆盖
  - 打包产物体积分布——bundle 体积、大依赖占比、重复打包
  - 兼容性风险——浏览器/系统兼容声明、polyfill 覆盖、CSS 厂商前缀
- 拆分依据：三类前端（Web/小程序/App）产出物形态一致，工具链接近（AST 解析/打包分析/路由扫描），可共用同一规范

**T3 遗留后端项目分析**：单体 / 微服务 / 数据管道共用

- 输入：代码仓库 + 项目形态参数
- 分析维度：
  - 基础维度（所有形态共有）：
    - 模块依赖图——模块间依赖方向、循环依赖、架构分层合规性
    - 数据库 Schema 演化历史——迁移脚本序列、回滚覆盖、Schema 与代码一致性
    - 配置层级——环境配置差异、硬编码值、敏感配置识别
  - 按形态启用附加维度：
    - 微服务形态：服务间调用拓扑、调用链超时配置、数据一致性策略、熔断降级覆盖
    - 数据管道形态：数据流向 DAG、各阶段吞吐量、异常数据去向、延迟瓶颈
- 拆分依据：三类后端产出物有共同基础维度 + 各自特有维度，可共用同一规范，通过形态参数启用附加分析

### 不再按语言拆分的理由

- 同一个"后端项目"标签下，Spring Boot 单体和分析一个微服务集群，关注的东西完全不同
- 前者需要 jdeps/ArchUnit 做模块依赖分析，后者需要链路追踪日志做拓扑分析
- 语言（Java/Go/Python）只影响选择哪个静态分析工具，不影响分析维度的定义——规范定义的是"产出什么"，不是"用什么工具产出的"
