## 1. 一句话定位

面向 AI Native 时代的人机协同分布式操作系统——以产研流程规范为抽象，意图驱动执行，对人和数字员工一视同仁。

## 2. 业务架构：11 域 + 17 聚合根（MVP）

### 2.1 域分类

| 分类 | 域 | 一句话职责 |
|------|-----|---------|
| **核心域（6）** | 经营决策域 | 商机→MRD→立项，AI 辅助可研 |
| | 项目管理域 | 项目全生命周期，资源调配 |
| | 项目实施域 | 任务执行与跟踪，声明 vs 证据验收 |
| | **人机协同域** | **OS 核心——流程规范定义 + Agent 原型/实例/运行时配置，一体两面** |
| | 技术资产域 | 可复用技术成果管理与推荐 |
| | 知识经验域 | 知识沉淀与 RAG 检索 |
| **支撑域（2）** | 用户权限域 | 统一身份体系（数字员工也是用户） |
| | 基础设施域 | Body 运行时、WebSocket 通信、LLM 路由 |
| **辅助域（2）** | 自我进化域 | 改进需求跟踪（v2） |
| | 价值评估域 | 积分计算与兑现（v2） |

### 2.2 上下文映射（5 种模式）

- **Customer-Supplier**：经营决策→项目（MRD 移交）、项目→实施（任务分解）、人机协同→实施（任务规范驱动分配）
- **Published Language**：用户权限域→所有域（统一身份 API）
- **Open Host Service**：基础设施域→人机协同域（Body 运行时 API）
- **Conformist**：自我进化域→所有域（被动接收事件）、个人域→所有域（只读聚合）
- **Anti-Corruption Layer**：人机协同域→外部 LLM API（模型路由抽象）、基础设施域→GitLab/Jira

跨域禁止直查数据库，走 API 契约或领域事件。

### 2.3 MVP 取舍

- ✅ 全部：经营决策、项目管理、项目实施、用户权限
- ⚠️ 简化：人机协同（基础 TaskType+Workflow+Agent 原型/实例，暂不做高级编排和自学习）、知识经验（扁平存储不分体系）、基础设施（Body 不进数据库，用 K8s ConfigMap）
- ❌ 后置：自我进化（只做埋点）、价值评估（只做硬编码积分）

## 3. 服务端架构：MVP 5 服务 → 成长 10 服务 → 成熟 12 服务

### 3.1 MVP 拓扑（5 个服务，Docker Compose 单机部署）

| 服务                 | 装载域                | 角色                                        | 端口   |
| ------------------ | ------------------ | ----------------------------------------- | ---- |
| **pdm-svc**        | 经营决策 + 项目管理 + 项目实施 | 核心业务聚合（三域高频联动，本地事务保证一致性）                  | 8080 |
| **capability-svc** | 数字员工 + 产研流程        | 流程规范定义 + Agent 原型/实例管理，改流程必牵扯改人，一体两面      | 8081 |
| **inf-svc**        | 基础设施域              | 运行时控制面——实例部署/WebSocket 通信/LLM 路由/Token 计量 | 8083 |
| **iam-svc**        | 用户权限域              | 统一身份与 RBAC 权限，所有服务底部承托                    | 8082 |
| **gateway-svc**    | —                  | 技术网关：路由/JWT 验签/限流/熔断/WS 升级                | 8085 |

**调用矩阵**：pdm/capability/inf 可同步调 iam-svc（鉴权）；核心域之间走 API 契约，支撑域禁止反向调核心域。

**异步事件总线**：RabbitMQ，at-least-once 投递。关键流——`MRD.Approved → Project.Created → Task.Assigned → Agent.TaskAssigned（WS 推送）→ Agent.TaskCompleted → Task.Completed`。

**拆分触发条件**：同时满足 ≥3 条（聚合根 >4 / 代码 >5 万行 / 团队 >5 人 / 跨域调用 >1000 次/月）才拆。

### 3.2 技术栈

- **框架**：Spring Boot 3.2.x + Spring Cloud 2023.0.x（Gateway + OpenFeign）
- **注册/配置**：Nacos 2.3.x
- **数据库**：PostgreSQL 16+（主数据库，JSONB 存半结构化，统一替代 MongoDB）
- **缓存**：Redis（会话、分布式锁、限流计数器）
- **事件总线**：RabbitMQ
- **链路追踪**：SkyWalking 9.x
- **网关**：Spring Cloud Gateway（JWT 验签 + Resilience4j 熔断 + Redis 限流）

### 3.3 演进路线

| 阶段 | 服务数 | 关键动作 | 架构特征 |
|------|--------|---------|---------|
| MVP（0-4 周） | 5 | 聚合核心域，本地事务 | Docker Compose |
| 成长期（4-24 周） | 10 | 拆 pdm-svc 为 4 服务，拆 capability-svc 为流程+Agent，引入 BFF + Saga | K8s + 分布式事务 |
| 成熟期（24 周+） | 12 | 每域独立部署，全面事件驱动 | K8s 多集群 + Istio + 事件溯源 |

## 4. 数字员工节点（ACP）：Rust 守护进程，7 模块

### 4.1 定位

Server 与本地 Agent 进程之间的协议适配器 + 本地监护人。不做业务判断，只做"翻译、路由、托管运行态"。

**节点 + 本地 Agent 进程 = Body**

### 4.2 7 模块职责

| 模块 | 一句话职责 |
|------|---------|
| ConnectionMgr | WebSocket 连接全生命周期（心跳/重连/Token 续期） |
| MessageRouter | 内存路由表，双向消息分发 |
| DriverRegistry | 3 种 Driver（Hermes/Claude Code/HttpApi）的统一调用入口 |
| ProtocolAdapter | Server JSON-RPC ↔ Agent 原生协议双向翻译 |
| ProcessManager | Agent 子进程生命周期（deploy/monitor/terminate） |
| ConfigManager | 配置优先级链：CLI > Server 覆盖 > 环境变量 > 配置文件 |
| SecretManager | MVP：Server 直下发明文凭证注入 Agent 环境；v2：Vault 引用模式 |

### 4.3 关键设计决策

- **实例生命周期 ≠ 任务执行**：先 deploy 实例，再向实例 assign 任务；一个实例可接收多个任务，避免冷启动开销
- **流式逐 chunk 转发**，不攒批
- **运行态不落盘**：崩溃后从 `sync.snapshot` 重建，不恢复本地快照
- **凭证不持久化**：注入 Agent 后立即释放内存引用

## 5. 客户端：应用端 + 管理后台（两个独立产品）

| 维度 | 应用端 | 管理后台 |
|------|--------|---------|
| 使用者 | 经营管理者/研发经理/AI 工程师 | AI Agent 架构师 |
| 功能 | 项目看板、任务执行、商机管理、工作台 | 原型管理、权限配置、节点监控、系统配置 |
| 交互 | 高频、强实时（看板/流式输出/SSE 推送） | 低频、配置为主（表格 CRUD + 表单） |
| 端形态 | PC/Web（MVP），后期 Electron/Tauri | PC/Web |

移动端规划中（聚焦审批 + 消息推送），MVP 不做。

## 6. 设计原则速查

1. **可逆性优先**——选容易回退的，不选"最优"的
2. **渐进式演进**——架构长出来不是设计出来
3. **域边界清晰**——跨域禁直查库，走 API 契约或事件
4. **OS 不自动决策**——立项/验收等关键节点必须人工确认
5. **人机一视同仁**——同一套任务规范，同一套验收标准
6. **声明 vs 证据**——只认可验证成果，不认口头承诺
7. **经验资产化的前提是结构化**——必须有元数据和场景关联
