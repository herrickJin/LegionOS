# 产研数字军团OS — 全景概要

> 一页纸看懂整个系统。本文档是 01-架构全景与设计原则 的精简版，供快速了解系统全貌。

---

## 1. 系统是什么

产研数字军团OS 是面向 AI Native 时代的产研全流程研发体系。本质是一个人机协同的分布式操作系统。

三个关键特征：

以产研流程规范为核心抽象。OS 定义什么角色在什么条件下执行什么动作、产出什么、如何验收——对人和数字员工一视同仁。

意图驱动的执行模式。人想清楚、定义问题、做决策；数字员工负责标准化执行、信息整合、流程推进。

经验资产化驱动价值。技术资产、知识条目、最佳实践在 OS 内持续沉淀与复用，形成组织复利。

---

## 2. 业务目标

支撑部门 AI Native 组织转型 — 四类角色与数字员工无缝配合，年内 50% 核心项目完成 AI Native 重构

应用软件部翻倍提效 — 需求交付周期缩短一倍，27 年 40% 人力支撑现有营业需求

协助部门经营模式转型 — 不增加人力完成自研产品交付，OS 作为对外业务的 AI 技术底座

---

## 3. 四类角色

经营管理者 — 全链路决策者。商机洞察、立项决策、经营分析、资源争取

复合型研发经理 — 经营与执行的枢纽。需求承接、任务分解、进度管控、交付收口

AI 工程师 — 执行角色。利用 AI 工具与数字员工完成任务、跟进前沿技术

AI Agent 架构师 — OS 建设者。OS 能力迭代、数字员工优化、技术资产运营

角色体系可扩展：当前聚焦内部四类职位，后续推广其他公司或面向 C 端用户时，通过 iam-svc 动态配置角色与权限，不硬编码。

---

## 4. 三层架构 + 两条通信通道

整体架构 = 三层 + 两条通信通道。

### 三层定位

Client 层：面向人的交互界面（应用端 + 管理后台），不直接访问微服务，统一经 API Gateway 接入

Server 层：业务逻辑的权威执行地，按域拆分微服务，所有业务数据以 Server 为权威源

数字员工节点层：Server 与 Agent 之间的通信桥梁，作为守护进程管理本地 Agent 的生命周期（启动/监控/停止），负责消息转发与协议适配。任务调度由 Server 负责，任务执行由 Agent 完成，节点不承担任何业务职责

### 两条通信通道

通道1（Client ↔ Server）：HTTPS + REST/GraphQL，经 Spring Cloud Gateway 统一鉴权与路由，JWT Token 验签

通道2（数字员工节点 ↔ Server）：WebSocket 长连接，Server 主动推送任务、节点主动上报执行结果。走标准 HTTP 端口与 Gateway，部署简单。节点断线后任务不丢失——Server 维护待推送任务队列，节点重连后拉取

## 5. 客户端产品矩阵（应用端 + 管理后台）

Client 层分两个独立产品，面向不同使用者，功能范围几乎不重叠。

### 应用端（经营管理者 / 复合型研发经理 / AI 工程师共用）

经营管理者 — 经营数据看板、商机洞察、立项审批、经营分析、资源概览

复合型研发经理 — 项目管理看板、需求承接与分解、任务分配辅助（人做决策）、进度监控（声明 vs 证据）、风险预警、交付收口

AI 工程师 — 个人工作台、任务执行与证据提交、数字员工协同（执行状态/结果确认/输出预览）、技术资产与知识查阅

共同模块 — 消息中心（通知/预警）、组织/项目切换、个人设置

技术形态：PC / Web（MVP），后期桌面端（Electron/Tauri）。通过 SSE 接收实时通知。

### 管理后台（AI Agent 架构师专用）

数字员工原型管理 — 原型 CRUD、版本管理、技能库

组织与权限管理 — 部门树维护、项目矩阵、角色配置、RBAC 功能权限分配、数字员工实例管理

系统配置 — LLM Provider 路由与 API Key、Token 配额、节点管理（注册/监控/激活码）、事件订阅规则

运维监控 — 节点在线状态、任务执行监控、审计日志

技术形态：PC / Web。交互以表格 CRUD + 配置表单为主，不追求高实时性。

### 移动端（规划中）

聚焦审批流程（立项审批、验收确认）和消息通知推送。不做完整的任务管理/项目管理。MVP 不做，成长期评估。

### 为什么应用端和管理后台是两个项目

目标用户不同、功能不重叠、权限模型差异大、迭代节奏不同。合并反而增加复杂度。

## 6. MVP 服务拓扑（4 个服务）

MVP 阶段采用聚合服务策略，4 个微服务：

pdm-svc（核心域聚合）— 经营决策 + 项目管理 + 项目实施。高频联动域聚合在一起，减少跨服务调用

asset-svc（资产聚合）— 技术资产 + 知识体系 + 自我进化 + 价值评估。文档类数据为主，与核心域变更频率不同

agent-svc（数字员工）— 数字员工原型定义、技能管理。原型包含角色定义、职责范围、技能清单、工具使用清单、初始记忆、运行环境定义、协议适配描述。原型会迭代，由本域管理

iam-svc（用户体系）— 类 EHR 系统。管理人类用户和数字员工实例的生命周期、组织层级（部门归属 + 项目矩阵）、RBAC 功能权限（菜单/按钮级）。数字员工也是用户，功能权限由本域管理，业务能力由数字员工域配置决定。数据权限（行级）由各业务域自行管理

基础设施层：Docker Compose 单机部署，PostgreSQL（主数据库）+ MongoDB（文档类资产）+ Redis（缓存/锁/去重）+ Nacos（注册/配置）+ RabbitMQ（事件总线）。向量库规划中。

iam-svc 底部承托其余 3 个服务，其他服务依赖 iam-svc 做用户鉴权与组织查询。

### 演进路线

MVP：4 个聚合服务（当前阶段）

成长期：按域拆分为 8 个服务（biz-svc、project-svc、process-svc、asset-svc、analytics-svc、agent-svc、iam-svc + 基础设施服务）

成熟期：12 个独立服务，每个域独立演进

## 8. 数字员工节点（Rust 守护进程）

数字员工节点是运行在终端上的 Rust 守护进程，核心职责：

管理本地 Agent 生命周期（启动/监控/停止）

消息转发与协议适配（Server ↔ Agent）

流式输出实时转发

节点连接维护（注册、心跳、断线重连）

支持的 Agent 类型：Hermes、Claude Code、其他 Agent。后期支持 Android / iOS 终端。

数字员工实例的创建流程：数字员工域定义原型 → 项目检索原型 → iam-svc 创建实例 → 基础设施层部署到节点 → 部署信息更新到 iam-svc

## 9. 架构设计原则

可逆性优先 — 优先选择容易回退的方案，AI Native 领域快速演进，今天的最优可能明天要推翻

渐进式演进 — 架构是长出来的不是设计出来的，每个阶段做每个阶段合适的事

域边界清晰 — 禁止跨域直接查库，跨域通信走 API 契约或领域事件

OS 不自动决策 — 关键节点（立项审批、任务验收）必须人工确认，AI 只推荐和执行

对人和数字员工一视同仁 — 同一套任务规范、同一套验收标准

声明 vs 证据 — OS 只认可验证的成果（Git Commit ID、文档 URL、测试报告），不认口头承诺

经验资产化的前提是结构化 — 散落的经验不是资产，必须有元数据和场景关联才能被复用
